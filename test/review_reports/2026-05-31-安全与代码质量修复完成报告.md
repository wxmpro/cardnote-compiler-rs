## CRITICAL 修复详情

### C1 — 异步 Mutex 阻塞 (api.rs)

**问题：** `std::sync::Mutex` 在 async 上下文中使用，阻塞 tokio worker 线程。

**修复：** `Arc<Mutex<usize>>` → `Arc<AtomicUsize>`

- 导入改为 `std::sync::{Arc, atomic::{AtomicUsize, Ordering}}`
- `lock().unwrap()` → `load(Ordering::Relaxed)` / `store(..., Ordering::Relaxed)`

**影响：** 并发场景下 tokio worker 不再被阻塞。

---

### C2 — 正则重复编译 (card_lint.rs)

**问题：** `fix_ref_format()` 每次 lint 调用重新编译 10+ 个正则表达式。

**修复：** 提取为 `std::sync::LazyLock` 预编译静态常量（共 9 个）

```rust
static RE_REF_PAGE: LazyLock<Regex> = LazyLock::new(|| {
    Regex::new(r"^(.+)_第(\d+)(?:-\d+)?页$").expect("硬编码正则")
});
```

**影响：** 正则编译成本从每次 lint 调用降至程序启动时一次。

---

### C3 — 路径遍历风险 (main.rs)

**问题：** `remove_dir_all(report_dir)` 执行前未验证路径前缀。

**修复：** 删除前用 `canonicalize` + `starts_with` 验证路径在输出目录内

**影响：** 防止恶意构造的 `report_dir` 导致任意目录删除。

---

### C4 — API key 文件权限 (config.rs)

**问题：** `~/.config/cardnote/.env` 创建后未设置权限，多用户系统可读取。

**修复：** 写入后立即 `chmod 600`（仅 Unix 系统）

**影响：** API key 文件仅所有者可读写。

---

### C5 — base_url 未验证 (api.rs)

**问题：** `--base-url` 参数未校验 URL 格式。

**修复：** 使用 `reqwest::Url::parse` 验证

**影响：** 防止 SSRF 和请求到非预期服务器。

---

## HIGH 修复详情

### H1 — 错误处理混用 (main.rs)

**问题：** `anyhow` 与 `AppError` 混用，错误链上下文丢失。

**修复：** `main.rs` 统一使用 `cardnote_compiler::error::Result`（即 `AppError`）

- `run()` 和 `handle_phase()` 返回类型改为 `cardnote_compiler::error::Result<()>`
- `anyhow::anyhow!(...)` → `AppError::Config(...)` / `AppError::TaskPanic(...)`

**影响：** 错误类型统一，调试时可精准匹配错误变体。

---

### H2 — 字符串匹配判错误 (main.rs:136)

**问题：** `e.to_string().contains("超时")` 判断错误类型，依赖本地化字符串。

**修复：** 模式匹配 `AppError::Timeout(msg)`

**影响：** 错误判断从字符串脆弱匹配升级为类型安全匹配。

---

### H3 — 字节 ≠ 字符 (pipeline.rs)

**问题：** `document.len()` 返回字节数，中文内容统计虚高 3 倍。

**修复：** 所有用户可见的字符统计改用 `document.chars().count()`

- 文档大小打印、CHUNK_SIZE 判断、ChunkInfo.size 均修正

**影响：** 字符统计准确，分块阈值判断与 CHUNK_SIZE 语义一致。

---

### H4 — API key 命令行暴露 (main.rs:42)

**问题：** `--api-key` 留在 shell history 和 `ps` 输出中。

**修复：** 添加帮助文本警告

```rust
/// API Key（会留在 shell history 中，建议优先使用环境变量 LLM_API_KEY）
```

**影响：** 用户在命令行传入 API key 时会看到风险提示。

---

### H5 — main.rs 上帝函数 (main.rs)

**问题：** `run()` 函数长达 394 行，承担全部职责。

**修复：** 拆分为 5 个独立函数 + 1 个参数结构体

| 函数 | 职责 |
|------|------|
| `run()` | 仅 CLI 解析与路由 |
| `handle_compile()` | 完整编译流程 |
| `handle_quality()` | Quality 命令 |
| `handle_scan()` | Scan 命令 |
| `handle_phase()` | Summary/Annotate/Cards/Graph |
| `RunArgs` | 阶段命令参数聚合 |

**影响：** 每个函数单一职责，可独立测试。

---

### H6 — 静默失败模式 (pipeline.rs)

**问题：** `compile_chunk()` 中子阶段失败后用空默认值继续。

**修复：** 改为显式错误传播 `?`

**影响：** 子阶段失败时由 `run_map_reduce` 重试或记录到 `failed_chunks`，数据完整性得到保障。

---

### H7 — 硬编码提供商配置 (providers.rs)

**问题：** 800+ 行硬编码 15 个提供商，新增需重编译。

**修复：** 支持从外部 `providers.json` 加载，硬编码作为 fallback

- 配置文件搜索路径：`~/.config/cardnote/providers.json` → `./providers.json`
- 新增 `export_builtin_config()` 导出默认配置

**影响：** 新增/修改提供商只需编辑 JSON 文件，无需重新编译。
