**审查角度**: 并发安全、竞态条件、边界条件、错误处理
**审查时间**: 2026-06-04
**审查范围**: cardnote-compiler-rs v0.1.49 全部源文件

## 用户场景确认

**用户明确说明**: 5000 本书逐本编译，不会批量生成。每次处理一本书。

**并发实际发生的环节**:
1. **Map-Reduce 内部**: 当一本书超过 `CHUNK_SIZE`(100K字符) 时，`pipeline.rs:718-741` 用 `Semaphore(max_workers=2)` 并行编译多个 chunk。这是程序内部的并发，与用户是否逐本处理无关。
2. **tokio::try_join!**: `pipeline.rs:1101-1108` 并发执行 summary + entities。

**因此，并发安全问题在逐本编译场景下依然存在，但影响范围限定在单本书内的 chunk 并行编译。**

## 一、并发安全

### [严重] 1. `rate_limiter.rs:35-48` — acquire() 在 tokio::time::sleep 期间持有 MutexGuard

```rust
// src/rate_limiter.rs
pub async fn acquire(&mut self) {  // &mut self 要求持有 MutexGuard
    loop {
        self.refill();
        if self.tokens >= 1.0 {
            self.tokens -= 1.0;
            return;
        }
        let wait_secs = needed / self.rate;
        tokio::time::sleep(Duration::from_secs_f64(wait_secs)).await; // 持锁 sleep!
    }
}

// 调用方 src/api.rs:443
limiter.lock().await.acquire().await;
```

**问题**: `acquire()` 需要 `&mut self`，所以 `MutexGuard` 在 `tokio::time::sleep` 期间被持有。如果 RPM 限制较低（如 10 RPM），sleep 可长达 6 秒。在此期间**所有其他 LLM 调用全部被阻塞**在 `limiter.lock().await` 上。

**在逐本编译场景下的实际影响**:
- 单本书（不分块）→ **无影响**（只有一个请求序列）
- 单本书分块编译（Map-Reduce, max_workers=2）→ **有影响**：两个并发 chunk 的 LLM 调用被完全串行化，失去并行加速

**修复建议**: 在 sleep 前释放锁：
```rust
pub async fn acquire(limiter: &Arc<tokio::sync::Mutex<RateLimiter>>) {
    loop {
        let mut guard = limiter.lock().await;
        guard.refill();
        if guard.tokens >= 1.0 {
            guard.tokens -= 1.0;
            return;
        }
        let wait_secs = (1.0 - guard.tokens) / guard.rate;
        drop(guard);  // ← 释放锁
        tokio::time::sleep(Duration::from_secs_f64(wait_secs)).await;
    }
}
```

### [中等] 2. `api.rs:82-104` — switch_to_next_model read-then-write 竞态

```rust
fn switch_to_next_model(&self) -> bool {
    let idx = self.current_model_idx.load(Ordering::Relaxed);  // 读
    let new_idx = idx + 1;
    self.current_model_idx.store(new_idx, Ordering::Relaxed);  // 写

```

**在逐本编译场景下的实际影响**:
- 单本书 → 无影响（只有一个 LLM 调用序列）
- Map-Reduce 分块编译 → 两个并发 chunk 同时触发 JSON mode fallback 时，可能重复切换到同一回退模型

### [中等] 3. `tests/common/mod.rs:19-28` — MockChat 在 async trait 中使用 std::sync::Mutex

```rust
pub struct MockChat {
    response: Mutex<String>,  // std::sync::Mutex，不是 tokio::sync::Mutex
}
```

**在逐本编译场景下的实际影响**: 测试单线程执行，不会触发。但如果用这些 mock 做并发测试，`std::sync::Mutex::lock()` 会在 async 上下文中阻塞 tokio worker。

tokio runtime 关闭时清理任务可能被中断，缓存文件积累。

## 二、错误处理

### [严重] 5. `pipeline.rs:1101-1108` — tokio::try_join! 短路丢失错误上下文

```rust
let (summary, entities) = tokio::try_join!(
    with_retry("摘要", || { generate_summary(...) }),
    with_retry("实体", || { extract_entities(...) }),
)?;
```

`try_join!` 在任一 future 失败时立即取消另一个。如果 entities 在第 3 次重试时失败，summary（可能在第 2 次重试后即将成功）被取消，两者错误信息都不可恢复。

**修复建议**: 使用 `tokio::join!` 收集两个 Result，分别处理：
```rust
let (summary_result, entities_result) = tokio::join!(...);
let summary = summary_result?;
let entities = entities_result.unwrap_or_default();
```

### [中等] 6. 整个代码库中广泛使用 `.ok()` 静默丢弃错误

| 文件 | 行号 | 被丢弃的错误 |
|------|------|-------------|
| `main.rs:35` | `let _ = std::fs::remove_dir_all(&path)` | 残留临时目录删除失败 |
| `main.rs:441` | `tokio::fs::create_dir_all(&doc_dir).await.ok()` | 文档备份目录创建失败 |
| `main.rs:446` | `tokio::fs::copy(&file, &dest).await.ok()` | 文档备份复制失败 |
| `main.rs:501` | `.unwrap_or(0)` | `ensure_book` 失败后 book_id=0 |
| `api.rs:408-416` | `let _ = std::fs::OpenOptions::new()...` | usage.log 写入失败（磁盘满） |

**在逐本编译场景下的实际影响**: 5000 本书 × 每本一次 = 累积的静默错误可导致备份不完整、编译记录丢失。特别是 `book_id=0` 的 `unwrap_or(0)` 可能使已处理的书籍被误认为"未处理"。

### [中等] 7. 三个 stage 文件的 LLM 调用失败统一转换为默认值

`stages/summary.rs:38-41`, `stages/entities.rs:35-39`, `stages/graph.rs:44-52` 的相同模式：
```rust
Err(e) => {
    eprintln!("    ⚠ {}", e);
    return Ok(Default::default()); // 静默降级
}
```

用户看到"编译成功"但不知道某些阶段返回了空数据。编译记录中的 `diagnostics.failures` 仅在 `run_single` 中记录，在 `compile_chunk` 中缺失。

### [轻微] 8. `config.rs:222` — get_provider_config 中 Box::leak 内存泄漏

每次调用 `get_provider_config` 都创建新的 `ProviderRegistry` + 重建约 15 个 Provider，然后 leak 3 个字符串。5000 本书 × ~100 bytes/次 ≈ 500KB 累积。

## 三、边界条件

### [中等] 9. `converter.rs:1012-1029` — is_likely_scanned 仅凭文件大小判断

```rust
fn is_likely_scanned(file_path: &str) -> bool {
    meta.len() > 2 * 1024 * 1024  // 2MB
}
```

认知科学和社科类 PDF 常带图表/复杂排版，1-3MB 很常见。仅凭文件大小 >2MB 判断为"扫描版"会导致超时估算失真（按每页90秒计算扫描版超时）。

### [中等] 10. `pipeline.rs:1006` — 短文档案（如诗集）分块后的边界处理

```rust
chunks.retain(|(_, doc)| doc.trim().len() > 200);
if chunks.is_empty() && !document.trim().is_empty() {
    chunks.push(("".to_string(), document.trim().to_string()));
}
```

当所有章节都短于 200 字符时，整本书被推入单个 chunk。如果该书超过 CHUNK_SIZE（如一本 15 万字符的诗集），单 chunk 会超出 LLM 上下文窗口。

### [轻微] 11. `pipeline.rs:718-722` — CARDNOTE_MAX_WORKERS 无上限校验

无上限约束。用户可能错误设置 1000 导致同时发起大量 LLM 请求。

### [轻微] 12. `dedup.rs:657` — 两张空内容卡片被判定为完全相同

```rust
if a.is_empty() && b.is_empty() { return 1.0; }
```

标题不同但内容都为空的卡片会被判定为"完全相同"而合并。

## 四、总结

| 严重等级 | 数量 | 关键项 |
|---------|------|--------|
| [严重] | 2 | rate_limiter 持锁 sleep; try_join! 错误丢失 |
| [中等] | 7 | Atomic 竞态; 错误静默吞; Mock Mutex; 文件大小误判 |
| [轻微] | 3 | Box::leak; worker 无上限; 空内容去重 |

**逐本编译场景下的影响评估**:
- **rate_limiter 持锁 sleep**: ❌ 有影响 — Map-Reduce 分块编译时并发被串行化
- **try_join! 错误丢失**: ❌ 有影响 — 分块编译时一个阶段失败取消另一个
- **Mock std::sync::Mutex**: ✅ 无影响 — 测试串行运行
- **Atomic 竞态**: ⚠️ 轻微 — Map-Reduce 分块编译时 JSON fallback 可能重复

**最需要修复的**:
1. rate_limiter 持锁 sleep（在 Map-Reduce 分块编译时影响并发性能）
2. try_join! 改为 join!（防止错误丢失）
3. 添加 `.ok()` 静默丢弃错误的诊断日志
