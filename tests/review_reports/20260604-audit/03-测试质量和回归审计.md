## 一、测试质量问题

### [严重] 1. mock_cards_json() 格式与真实 LLM 输出不匹配 — 所有卡片集成测试跑降级路径

`tests/common/mod.rs:152-176` 的 `mock_cards_json()` 返回 `{"cards": [...]}` JSON 结构。

`generate_cards` 有两个策略路径：
- **extract-then-assign**: Step1 期望 `{"knowledge_points": [...]}`，Step2 期望 `#卡片类型\n标题：...` 文本格式
- **legacy**: 期望 `---` 分隔 + `标题：` 字段 + `#卡片类型名` 标签的文本格式

两者都无法解析 `{"cards": [...]}` 格式。

**影响**: 以下测试实际跑的是"格式不匹配→回退→返回空列表"的降级路径，从未验证卡片解析成功路径：
- `integration_stages.rs` — `test_generate_cards_with_valid_json`
- `integration_stages.rs` — `test_full_pipeline_simulation` (阶段3)
- `integration_stages.rs` — `test_generate_cards_with_empty_response`
- `integration_stages.rs` — `test_generate_cards_assign_unique_ids`
- `e2e_pipeline.rs` — `test_e2e_four_stage_pipeline` (阶段3)

**修复**: 提供两套 mock 响应，分别匹配 extract-then-assign 和 legacy 格式。

### [严重] 2. `integration_stages.rs:289` — 恒真断言

```rust
assert!(result.is_empty() || !result.is_empty()); // 不会 panic
```

这是逻辑重言式（P ∨ ¬P），无论 result 是什么永远为 true。此断言验证了零个属性。

**修复**: 删除此断言，替换为 `assert!(result.is_ok())` 或验证空响应的特定行为。

### [中等] 3. `integration_stages.rs:551` — test_summary_recovery_on_prompt_load_failure 名不副实

测试名声称在测 "prompt load failure"，但 `mock_load_prompt` 能正常处理 `"summary"`。测试从未走到 prompt 加载失败的代码路径。

**修复**: 传入一个 `mock_load_prompt` 无法处理的 prompt 名称，或改名并补充真正的失败测试。

### [中等] 4. `integration_stages.rs:510-531` — test_full_pipeline_simulation 的卡片-实体关联检查是空循环

当 `cards` 为空时（因为 mock 格式不匹配），循环体完全跳过。没有断言验证关联关系。

### [中等] 5. `e2e_pipeline.rs:93-104 + 318-353` — 标题唯一性、UUID 检查仅在 cards 非空时执行

`if !dedup_result.cards.is_empty()` 保护下的断言几乎从不执行（因为 cards 几乎总是空）。

### [中等] 6. `e2e_pipeline.rs:360-377` — 错误恢复测试容忍两种相反结果

```rust
match result {
    Ok(entities) => assert!(entities.is_empty()),
    Err(_) => {},  // 不做任何断言：如果行为从 Ok 变成 Err，测试仍通过
}
```

## 二、测试盲区

### [严重] 7. 跨书去重隔离 — 零测试覆盖

用户核心需求：同一本书内去重，不同书不去重。`semantic_dedup` 在当前代码中的调用约定正确（每次编译一本书时独立调用），但**没有任何测试验证跨书隔离**。

如果未来的代码变更错误地在跨书合并时调用了 `dedup_cards`，现有测试无法发现。

**需要添加的测试**:
```
Book A: 卡片"系统思维", content="系统思维是一种整体性方法。"
Book B: 卡片"系统思维", content="系统思维强调反馈循环和涌现。"
验证: 两次独立编译后，各自的卡片保持独立（不被合并）
```

### [严重] 8. SQLite 持久化 — 零测试覆盖

`src/batch.rs` 的 `CompileTracker` 有完整的事务管理、外键约束、schema 迁移，但是：
- 零测试验证 `record_compilation` 写入后可通过 `recent()` 读取
- 零测试验证外键约束（ON DELETE CASCADE）
- 零测试验证 `ensure_book` 的 UPSERT 逻辑
- 零测试验证 schema v1→v2 迁移
- 零测试验证 `format!("{:?}", card.card_type)` 的格式一致性

**需要添加的测试**:
1. `record_compilation` → `recent()` 读写往返
2. `ensure_book` 重复调用（UPSERT）
3. `stats()` 聚合查询正确性
4. `mark_reviewed()` 状态更新

### [严重] 9. Map-Reduce 路径 — 零测试覆盖

`Pipeline::run_map_reduce` 是 ~200 行的核心代码路径，包含语义分块、并行编译、断点续编译、失败重试、Reduce 合并。零测试覆盖。

**需要添加的测试**:
1. `semantic_chunk` 分块逻辑（按标题分块、chunk 不跨章节边界）
2. `CompileCache` 读写和失效
3. `merge_doc_summaries` 本地合并（LLM 不可用时的回退）
4. `extract_overlap` 重叠内容提取

### [中等] 10. legacy 卡片生成路径 — 未测试全部 12 种卡片类型

`parse_single_type_cards` 只有单元测试覆盖了 Term 和 Knowledge 两种类型。其余 10 种类型（Person/Quote/Action/Event/Graph/NewWord/Note/Index/CounterIntuit/Review）的解析逻辑未被集成测试覆盖。

### [中等] 11. Stage 缓存 — 零测试覆盖

`pipeline.rs` 中 `StageCache` 的缓存命中/未命中、文档哈希变化失效、prompt 模板变化失效、模型变化失效——全部零测试。

### [中等] 12. FNV-1a 哈希 golden value — 未设置

`pipeline.rs` 使用 FNV-1a 替代 `DefaultHasher` 的动机是"Rust 版本升级后不变化"。但当前测试没有用 golden value 做交叉验证。如果 FNV-1a 实现被意外修改，现有测试无法发现。

**修复**: 添加 golden value 断言：
```rust
assert_eq!(fnv1a_hash_str("hello"), "a430d84680aabd0b");
```

### [中等] 13. 路径安全 — 缺少符号链接遍历测试

`integration_cli.rs` 测试了 `..` 路径遍历被拒绝。但 `std::fs::canonicalize` 会跟随符号链接。没有测试验证指向 `/etc/passwd` 等敏感位置的符号链接行为。

### [轻微] 14. 12 种卡片类型 to_markdown 各路径 — 部分覆盖

`test_all_card_types_to_markdown` 测试了每种类型生成 markdown 不 panic，但未验证特定类型的特殊字段（如 Person 卡是否有 biography 字段需要呈现、Action 卡是否有 steps 字段）。
