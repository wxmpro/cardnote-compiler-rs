# CardNote Compiler (Rust) — 技术债与风险评估

> **评估日期**：2026-05-31
> **评估范围**：v0.1.23，~25,000 行 Rust，30 个源文件，20+ 个 Git 版本
> **评估方法**：全量源码阅读 + 架构分析 + 版本演化回溯

---

## 风险等级定义

| 等级 | 含义 | 行动原则 |
|------|------|---------|
| 🔴 高 | 可导致崩溃、数据丢失、安全漏洞或功能静默失败 | 应在下一版本修复 |
| 🟡 中 | 影响可维护性、性能或边缘情况正确性 | 应纳入近期迭代 |
| 🟢 低 | 代码质量、一致性、未来兼容性问题 | 可在重构窗口处理 |

## 🔴 高风险项（4 项）

### R1. Anthropic / Gemini 协议未真正实现——请求端点硬编码为 OpenAI 格式

- **位置**：`src/api.rs:321`（`send_request` 的 URL 拼接）
- **严重程度**：🔴 高
- **类型**：功能缺陷 / 协议实现不完整

**当前行为**：
`send_request()` 固定使用 `/chat/completions` 端点（OpenAI 兼容格式），无论 Provider 声明的协议类型是什么：

```rust
// api.rs:321
let url = format!("{}/chat/completions", self.base_url.trim_end_matches('/'));
```

**问题**：
- `providers.rs` 中 Anthropic 标记为 `ProviderProtocol::Anthropic`，其真实 API 端点是 `https://api.anthropic.com/v1/messages`，不是 `/chat/completions`
- Gemini 标记为 `ProviderProtocol::Gemini`，端点是 `https://generativelanguage.googleapis.com/v1/models/{model}:generateContent`
- 如果用户直连 Anthropic API（非通过代理中转），会收到 HTTP 404
- 目前能"正常工作"仅因为用户可能通过兼容代理（如 ZSCC、Linux.do Hub）访问这些模型

**影响范围**：
- 所有使用 `ProviderProtocol::Anthropic`、`ProviderProtocol::Gemini`、`ProviderProtocol::Cohere` 的直连场景
- `health.rs` 中的健康检测同样受影响（使用同一个 `LlmClient`）

**建议修复**：
1. 在 `send_request()` 中根据 `self.provider_id` 查 `ProviderRegistry` 获取协议类型
2. 为 Anthropic 实现 Messages API 端点 + 请求体格式转换
3. 为 Gemini 实现 generateContent 端点
4. 或者：在 `create_client()` 阶段对非 OpenAI 兼容协议直接返回明确错误，避免静默失败

**相关版本**：此问题自 v0.1.18（引入多协议支持）以来一直存在。

### R2. Python 子进程依赖过重 + 无统一超时兜底

- **位置**：`src/converter.rs`（多处 `Command::new("python3")`）
- **严重程度**：🔴 高
- **类型**：可靠性 / 可移植性

**当前行为**：
文档转换链路依赖多个 Python 子进程，部分有超时、部分没有：

| 调用位置 | 用途 | 超时控制 |
|---------|------|---------|
| `read_pdf_raw()` | PyMuPDF 提取文本 | ❌ 无超时 |
| `read_markitdown()` | MarkItDown 转换 | ❌ 无超时 |
| `read_pdf_scan()` | MinerU OCR | ❌ 无超时（函数内部无 tokio::timeout） |
| `read_pdf_scan_fallback_mineru()` | MinerU OCR fallback | ❌ 无超时 |
| `read_pdf_expert_ocr()` | PDF Expert OCR | ❌ 无超时 |
| `split_pdf_by_toc()` | PyMuPDF TOC 拆分 | ❌ 无超时 |
| `save_pdf_range()` | PyMuPDF 页面保存 | ❌ 无超时 |
| `convert_to_markdown_async_with_timeout()` | 顶层包装 | ✅ 有超时（但只包裹了 spawn_blocking） |

**问题**：
- `convert_to_markdown_async_with_timeout()` 的超时只包裹了 `spawn_blocking`，内部的 Python 子进程可能远超超时时间仍在运行（僵尸进程）
- 如果一个 PDF 让 PyMuPDF 陷入无限循环（已知的 PyMuPDF bug），整个编译流程永久挂起
- 没有统一的子进程管理机制——无法强制 kill 超时的子进程

**建议修复**：
1. 为所有 Python 子进程调用添加 `tokio::time::timeout` 包装
2. 超时后执行 `child.kill()` 确保子进程被终止
3. 统一抽取一个 `run_python_with_timeout(script: &str, timeout: Duration) -> Result<String>` 工具函数

### R3. LLM JSON 输出无反序列化 Schema 校验——静默字段丢失

- **位置**：`src/stages/cards.rs`、`src/stages/entities.rs`、`src/stages/graph.rs`、`src/stages/summary.rs`
- **严重程度**：🔴 高
- **类型**：数据完整性

**当前行为**：
所有 Stage 的 LLM 返回处理流程是：
1. `chat_json()` 返回 `serde_json::Value`
2. 手工从 `Value` 中提取字段：`value.get("title").and_then(|v| v.as_str())`
3. 如果字段缺失，使用默认值（空字符串 / 空数组）继续

**问题**：
- LLM 输出格式如果发生微小变化（如字段名从 `card_type` 变成 `type`），整个提取静默返回空值
- 没有机制检测 "本次 LLM 返回了 10 张卡片但只成功解析了 3 张"——被静默丢弃的卡片无任何日志
- 当前的"安全"行为（失败返回默认值）掩盖了真正的质量问题

**实际影响场景**：
- v0.1.19 修复了金句卡 `source` 字段缺失——说明 LLM 确实会漏字段
- v0.1.18-patch3 修复了动态 `max_tokens` 导致 JSON 截断——截断的 JSON 解析失败后，整个块的卡片全部丢失

**建议修复**：
1. 为每个 Stage 的输出定义严格的 `#[derive(Deserialize)]` 结构体
2. 解析时记录缺失字段的警告
3. 增加 "解析成功率" 指标：`解析到的卡片数 / LLM 声称的卡片数`
4. 解析失败率超过阈值时触发诊断报告

### R4. `scan_directory_async` 和 `scan_directory` 代码高度重复

- **位置**：`src/scan.rs:54-127` vs `src/scan.rs:130-235`
- **严重程度**：🔴 高（维护风险）
- **类型**：代码重复 / DRY 违反

**当前行为**：
两个函数实现完全相同的业务逻辑（扫描 PDF 文件、检测文字层、分类统计），唯一区别是：
- `scan_directory` 是同步顺序执行
- `scan_directory_async` 用 `tokio::spawn_blocking` 并行执行

**代码重复量**：约 120 行几乎相同的代码。

**问题**：
- 修改检测逻辑需要改两处——已在 v0.1.23 发生过一次：PDF 质量评分逻辑只在 `make_scan_result` 中更新，但两处的分类和打印逻辑各自维护
- 两处的打印输出格式略有差异，已经出现不一致的苗头

**建议修复**：
抽取共享核心逻辑 `scan_directory_impl(pdf_files, threshold) -> Vec<ScanResult>`，两个函数只负责调度方式（同步 vs 并行）和输出打印。或者直接废弃同步版本，统一使用异步版本（`scan_directory` 内部调用 `tokio::runtime::Handle::current().block_on(async {...})`）。

## 🟡 中风险项（5 项）

### M1. `CompileConfig` 定义但未被 Pipeline 实际使用

- **位置**：`src/doc_type.rs:33-72`（定义） vs `src/pipeline.rs:429`（实际使用）
- **严重程度**：🟡 中
- **类型**：死代码 / 设计未落地

**当前行为**：
`DocTypeDetector` 为每种文档类型精心定义了不同的编译策略：

| 文档类型 | chunk_size | max_cards_per_chunk |
|---------|-----------|-------------------|
| Book    | 50000     | 15 |
| Paper   | 30000     | 12 |
| Article | 15000     | 8  |
| Report  | 40000     | 10 |
| Manual  | 35000     | 10 |

但 `pipeline.rs:429` 的分块判断直接使用全局常量：

```rust
if document.chars().count() <= CHUNK_SIZE {  // CHUNK_SIZE = 50000，固定值
    self.run_single(...)
} else {
    self.run_map_reduce(...)
}
```

且 `max_cards_per_chunk` 完全没有被引用。

**影响**：
- 论文（实际只需要 30000 字符分块）被当作书籍处理，可能导致 LLM 上下文过载
- 文章（15000 字符分块）被过度合并，丢失细粒度
- 文档类型检测的工程投入（200+ 行特征提取 + 评分逻辑）产出为零

**建议修复**：
1. 在 `Pipeline::run()` 中将 `doc_detection.doc_type.compile_config()` 的 `chunk_size` 传给分块逻辑
2. 如短期内不打算使用，至少添加 `#[allow(dead_code)]` 注释说明原因

### M2. 并发模型声明与实现不一致

- **位置**：`src/config.rs:20-21` vs `src/pipeline.rs:667`
- **严重程度**：🟡 中
- **类型**：注释/代码不一致

**当前行为**：

```rust
// config.rs:18-21
/// 串行编译（避免触发 API 限流）
/// [M4] 实际为串行，注释已修正以反映真实行为
pub const MAX_WORKERS: usize = 1;

// pipeline.rs:667
let semaphore = Arc::new(Semaphore::new(2));  // 实际并发数 = 2
```

**问题**：
- `MAX_WORKERS` 常量（=1）完全没有被使用——已经有一个 `[M4]` 注释承认了注释与代码不符，但常量本身没被清理
- Map 阶段实际的并发控制是通过 `Semaphore::new(2)` 实现
- 如果有开发者基于 `MAX_WORKERS` 做判断，会得出错误结论

**建议修复**：
删除 `MAX_WORKERS` 常量或将其值改为 2 并实际使用它来初始化 Semaphore。

### M3. Stage 缓存策略激进而无 Prompt 版本管理

- **位置**：`src/pipeline.rs:138-199`（`StageCache`）
- **严重程度**：🟡 中
- **类型**：缓存失效策略

**当前行为**：
Stage 缓存的 key = `FNV1a(stage + document_hash + prompt_hash + model)`。

**问题**：
- Prompt 模板（`prompts/*.md`）的微小修改就会导致缓存全部失效——这不是 bug，但缺少提示机制告知用户"因为 prompt 变了，缓存已失效"
- `StageCache::VERSION` 硬编码为 `"v1"`，升级时需手动修改——容易遗忘
- 缓存文件清理由 `cleanup_cache_dir()` 在 `Pipeline::new()` 时触发（30 天过期 + 200 文件上限），但如果用户一天内多次编译不同文档，缓存可能超过 200 文件仍未被清理

**建议修复**：
1. 在缓存失效时打印一条 `💾 Stage 缓存因 prompt 变更而失效，将重新调用 LLM` 消息
2. 将缓存版本绑定到 `Cargo.toml` 的版本号
3. 在每次 `save` 时检查总文件数（而非仅在 Pipeline 构造时）

### M4. 文件大小上限 `MAX_FILE_SIZE_MB = 500` 对大 PDF 是理论值，实际会在转换层先 OOM

- **位置**：`src/config.rs:140` + `src/converter.rs:39-48`
- **严重程度**：🟡 中
- **类型**：OOM 防护缺口

**当前行为**：
- `MAX_FILE_SIZE_MB = 500`——即允许处理 500MB 的文件
- 但 `converter.rs` 中的 `check_file_size()` 只在入口检查文件大小
- 对于 PDF：MarkItDown 或 PyMuPDF 子进程会将整个 PDF 的文本提取到内存
- 一个 500MB 的 PDF 提取出的文本可能远超可用内存

**实际上的保护**：
- `read_pdf_split()` 对 >100MB 的 PDF 跳过整体转换，按 TOC 拆分
- 但 `read_pdf_single()` 仍然在第一层尝试 MarkItDown 整体转换

**建议修复**：
1. 将 `MAX_FILE_SIZE_MB` 降至 200 或添加 "超过 100MB 强制拆分" 的逻辑到 `read_pdf_single` 入口
2. 对文本文件增加流式读取（`read_text_streaming` 已实现但阈值是 `MAX_TEXT_FILE_STREAM_MB = 50`）

---

### M5. 书名提取逻辑脆弱——经历过 3 次修复才稳定

- **位置**：`src/converter.rs:1065-1159`（`extract_pdf_metadata` + `looks_like_page_marker`）
- **严重程度**：🟡 中
- **类型**：启发式规则脆弱性

**当前行为**：
书名提取策略历经 3 个版本迭代（v0.1.12 ~ v0.1.14）才基本稳定：
1. v0.1.12 前：从文本内容中搜索——误报率极高
2. v0.1.12：修复页码标记污染
3. v0.1.13：修复更多误判模式
4. v0.1.14：彻底删除文本搜索，只保留 PDF Info 字典 + 文件名

**残留问题**：
- `looks_like_page_marker()` 有 5 条启发式规则，覆盖了已知误判模式，但新增误判模式（如中文竖排书）可能仍需补充
- 没有单元测试覆盖 `looks_like_page_marker` 的各个分支
- `extract_pdf_metadata` 有测试但 `looks_like_page_marker` 没有

**建议修复**：
为 `looks_like_page_marker` 添加边界测试用例，覆盖所有 5 条规则。

## 🟢 低风险项（5 项）

### L1. `BookCompilationResult` 模型定义但从未使用

- **位置**：`src/models.rs:454-467`
- **严重程度**：🟢 低
- **类型**：死代码

`BookCompilationResult` 及其关联的 `ChapterPlan` 模型定义了完整的书籍级编译结果结构，但：
- `pipeline.rs` 只使用 `CompilationResult`
- `output.rs` 的 `save_book()` 函数接受的是分离的参数而非 `BookCompilationResult`
- 说明多文档策展功能可能在设计阶段被考虑但未实现

**建议**：如果确认不再需要，删除以减小编译单元大小；如果未来需要，添加 `#[allow(dead_code)]` + TODO 注释。

### L2. 测试覆盖不均衡

- **位置**：整个 `src/` 目录
- **严重程度**：🟢 低（不影响功能，但影响重构信心）
- **类型**：测试债务

**覆盖较好的模块**：
- `models.rs` — Mermaid 转义、Card 序列化、Summary 解析（20+ 测试）
- `converter.rs` — 编码检测、文件名清理、流式读取（15+ 测试）
- `dedup.rs` — 语义去重、shingle 计算、质量评分（15+ 测试）
- `doc_type.rs` — 5 种文档类型检测 + CompileConfig 校验（15+ 测试）

**覆盖不足的模块**：
- `pipeline.rs` — Map-Reduce 流程无集成测试（只有单元测试覆盖分块/去重/合并的辅助函数）
- `api.rs` — 无任何测试（LLM 客户端、JSON 重试、模型回退）
- `stages/` — 摘要解析有测试，但 Entity/Cards/Graph 的解析无测试
- `scan.rs` — PDF 文字层检测无测试
- `health.rs` — Provider 健康检测无测试

**建议**：优先为 `pipeline.rs` 的 Map-Reduce 流程添加集成测试（用 mock LLM 客户端）。

### L3. `converter.rs` 的 `read_pdf_scan()` 和 `read_pdf_scan_fallback_mineru()` 存在大量重复代码

- **位置**：`src/converter.rs:568-644`
- **严重程度**：🟢 低
- **类型**：代码重复

两个函数的 MinerU 调用逻辑完全相同（启动命令、参数、输出文件查找），差异仅在：
- `read_pdf_scan()` 多了 PDF Expert OCR fallback 尝试
- `read_pdf_scan_fallback_mineru()` 被调用时已经确定"PDF Expert 不可用"

约 60 行重复代码。

**建议**：抽取 `run_mineru_ocr(file_path) -> Result<String>` 公共函数。

### L4. `resolve_book_title()` 逻辑分散在两个模块

- **位置**：`src/main.rs:218-230` + `src/converter.rs:1065-1108`
- **严重程度**：🟢 低
- **类型**：职责分散

书名解析逻辑分布在：
- `main.rs::resolve_book_title()` — 编排 PDF 元数据提取 + 文件名 fallback
- `converter.rs::extract_pdf_metadata()` — PDF Info 字典解析

此外 `converter.rs::guess_title()` 提供了另一种标题猜测方式（从 Markdown 标题），但 `resolve_book_title` 没有使用它。

**建议**：将 `resolve_book_title` 移入 `converter.rs` 并整合 `guess_title`，使 `main.rs` 只负责调用。

### L5. Export 功能使用 `Box::leak` 有意内存泄漏

- **位置**：`src/config.rs:228-231`
- **严重程度**：🟢 低（泄漏量极小）
- **类型**：内存泄漏（有意）

```rust
pub fn get_provider_config(provider: &str) -> Option<ProviderConfig> {
    let registry = ProviderRegistry::new();
    let p = registry.find_by_alias(provider)?;
    let default_model = p.default_model()?;
    Some(ProviderConfig {
        provider: Box::leak(p.id.clone().into_boxed_str()),   // 故意泄漏
        base_url: Box::leak(p.default_base_url.clone().into_boxed_str()),  // 故意泄漏
        default_model: Box::leak(default_model.id.clone().into_boxed_str()), // 故意泄漏
    })
}
```

每次调用泄漏 3 个字符串（~100 字节），调用频率极低（每个 `cardc` 进程仅 1-2 次），实际影响可忽略。但这种模式是一种"为了满足生命周期而妥协"的信号——`ProviderConfig` 使用了 `&'static str` 而非 `String`。

**建议**：将 `ProviderConfig` 的字段改为 `String`，消除 `Box::leak`。

## 架构健康度总评

**正面**（做得好的地方）：

- **错误处理体系完善**：12 种错误类型覆盖了所有已知失败模式；编译诊断报告（`CompilationDiagnostics`）能清晰记录哪个阶段失败、为什么、重试了几次
- **防御性 LLM 编程三层模式**：API 层 JSON 重试（3 次指数退避）→ Stage 层重试（with_retry）→ 模型自动回退（switch_to_next_model），每层解决不同粒度的问题
- **Provider 注册表可外部扩展**：支持通过 JSON 文件热加载新 Provider，无需重编译
- **文档类型检测设计完善**：特征提取 + 多维度评分 + 置信度计算——虽然暂未激活使用
- **语义去重算法**：字级 shingle + Jaccard 相似度 + 连通分量合并 + 质量评分驱动的 canonical 选择，适合中文无需分词

**需关注**：

- **协议层是最大的单点风险**：如果用户绕过代理直连 Anthropic/Gemini，整个系统对这两个 Provider 完全不工作
- **Python 依赖是运维脆弱性的集中点**：8 处 Python 子进程调用，任何一处环境不一致都会导致功能降级
- **测试覆盖的"空心化"**：辅助函数测试充分，但核心流程（Map-Reduce、LLM 调用）缺少集成测试

**总体评估**：项目已从"原型验证"阶段进入"生产化"阶段。核心架构设计合理，但在协议完整性、子进程可靠性和测试覆盖三个维度有明确的改进空间。14 项技术债中 4 项高风险，建议优先解决协议和超时问题。
