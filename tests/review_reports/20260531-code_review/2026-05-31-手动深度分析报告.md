## 维度 2: 实现

### 2.1 核心数据流

```
PDF/文档 → [converter.rs] → Markdown → [quality::analyze] → 质量评分
                                              ↓ (≥60分)
                          [pipeline.rs:run] → 单轮 / Map-Reduce
                                              ↓
                              ┌───────────────┼───────────────┐
                              ↓               ↓               ↓
                         [summary]      [entities]      [cards]
                              └───────────────┼───────────────┘
                                              ↓
                                         [graph]
                                              ↓
                                    [dedup.rs] 去重
                                              ↓
                                    [output.rs] 文件输出
```

### 2.2 关键实现缺陷

#### 🔴 缺陷 1: `document.len()` 字节数 bug（影响中文文档卡片规划）

**位置**: `src/stages/cards.rs:248`

```rust
let plan = CardPlanner::plan(doc_type, document.len());  // ← 字节数！
```

**根因**: `CardPlanner::plan_book` 使用 `char_count > 50000` 判断 scale。对于中文文档，`len()`（UTF-8 字节）≈ 3× `chars().count()`，导致短中文文档被误判为小规模，卡片数量规划偏少。

**影响**: 纯中文书籍（如《聪明的阅读者》）的卡片生成量可能比预期少 2/3。

**修复**: 改为 `document.chars().count()`。

---

#### 🔴 缺陷 2: 编译失败时静默跳过，不返回错误

**位置**: `src/pipeline.rs:449-462`（及所有 4 个阶段）

```rust
let summary = match generate_summary(...) {
    Ok(s) => { /* 正常 */ }
    Err(e) => {
        diagnostics.failures.push(StageFail { ... });
        Summary::default()  // ← 返回空值，不中断编译
    }
};
```

**根因**: 所有阶段失败都返回默认值，记录到 `diagnostics`，但不中断编译。

**影响**: 如果 LLM API 不稳定，工具输出"编译完成"但目录内是空文件。这是**"输出丢失"最可能的根因**。

**修复方向**: 在 `main.rs` 增加编译结果健康检查，空结果时给出明确警告。

---

#### 🟠 缺陷 3: 外部命令注入风险

**位置**: `src/converter.rs` 多处（`read_pdf_raw`, `read_pdf_ocr_fallback` 等）

```rust
Command::new("python3")
    .arg("-c")
    .arg(r###"...{file_path}..."###)  // file_path 直接嵌入字符串
```

**根因**: `file_path` 通过字符串拼接嵌入 Python 代码。如果文件名包含 `"` 或换行符，可能导致 Python 字符串截断/注入。

**修复**: 使用 `std::process::Command::arg()` 安全传入参数，而非字符串拼接。

---

#### 🟠 缺陷 4: 卡片解析对 LLM 输出格式脆弱

**位置**: `src/stages/cards.rs:339-458`

`extract_field` 使用正则匹配字段前缀（如 `标题：`），如果 LLM 输出加粗格式（`**标题：**`），字段提取失败。

---

#### 🟡 缺陷 5: Map-Reduce 失败块处理不够明显

**位置**: `src/pipeline.rs:700-710`

失败块的重试后仍然失败时，只打印 stderr 警告，但继续创建输出目录。用户可能误以为编译成功。

---

### 2.3 "输出丢失"根因分析

| 可能性 | 根因 | 排查方法 |
|--------|------|---------|
| **1** | LLM 阶段失败 → 空结果 → 仍创建目录 | 检查 output 下是否有空的时间戳目录 |
| **2** | 失败率 >20% → 整体中断 | 检查终端是否有"分块编译失败率"输出 |
| **3** | API key 失效 → 全部失败 | 检查 `cardc doctor` 输出 |
| **4** | 输出目录内容为空 → 被忽略 | `ls -la output/` 查看 |

## 维度 4: 可靠性

### 4.1 重试策略

| 层级 | 位置 | 策略 | 最大重试 |
|------|------|------|---------|
| API 层 | `api.rs:206` | HTTP / JSON 解析失败 | 3 次（指数退避 2→4→8s） |
| JSON 层 | `api.rs:142` | 模型不支持 JSON mode | 自动 fallback 模型 |
| Stage 层 | `pipeline.rs:939` | 单阶段失败 | 3 次 |
| Chunk 层 | `pipeline.rs:651` | 分块编译失败 | 1 次整体重试 |

**问题**: `run_single`（单轮编译）**没有 Stage 层重试**，只有 Map-Reduce 模式有。

### 4.2 缓存策略

- **CompileCache**: 分块级，支持断点续编
- **StageCache**: 阶段级，按 `(stage + 文档哈希 + prompt 哈希 + 模型)` 失效
- **问题**: 无清理机制，`.cardc_cache/` 可能无限增长

## 维度 6: 性能
