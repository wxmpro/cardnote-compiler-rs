# LLM 调用链路优化讨论

> **日期：** 2026-05-31  
> **项目：** cardnote-compiler-rs v0.1.17  
> **目标：** 为 PDF 生成高质量卡片——从 LLM 调用角度找优化空间  
> **状态：** 讨论完成，进入代码改造阶段  

---

## 一、当前架构速览

单次编译的 LLM 调用链路：

```
run_single() ──→ generate_summary()  ────────→ 1 次 (文本)
            ├──→ extract_entities()  ────────→ 1 次 (JSON)
            ├──→ generate_cards()  ──────────→ N 次 (文本，按类型)
            │      ├── knowledge_card
            │      ├── term_card
            │      ├── review_card
            │      └── counter_intuit_card
            └──→ build_graph()  ─────────────→ 1 次 (JSON)
```

单文档 **7 次** LLM 调用。3-chunk 书籍 **~22 次** LLM 调用。

## 二、八大问题诊断

### 问题 1：卡片生成的 N+1 查询

`generate_cards()` 为每种卡片类型单独调用一次 LLM。同一份文档内容被重复送入 LLM 多次，造成 input token 冗余。

**分析结论：** 保留分类型专用 prompt 的设计（保障质量），通过 Prompt Caching 解决重复 input 问题（详见第三节讨论）。

---

### 问题 2：完全未利用 Prompt Caching

`send_request()` 每次发送完整 messages 数组，没有任何 caching 标记。Anthropic/OpenAI 的 Prompt Caching 可节省 70-90% 重复 input token。

**改造计划：** 在 `LlmMessage` / `LlmRequest` 中增加 cache 控制字段，按 ProviderProtocol 分支实现。不支持的 Provider 自动 fallback（详见用户 Provider 现状）。

---

### 问题 3：并发策略过度保守

`MAX_WORKERS = 1`，chunks 串行处理。

**改造计划：**
- 同一 chunk 内：`summary` 与 `entities` 并发（互不依赖）
- 不同 chunk 间：最多 2 个并发（Semaphore 控制）
- `cards` 和 `graph` 串行（依赖 entities 结果）

预期 wall-clock 时间减少 40-60%。

---

### 问题 4：零 Token 用量追踪

`extract_content()` 丢弃了响应中的 `usage` 字段。无法分析成本、无法定位瓶颈。

**改造计划：** 新增 `LlmUsage` 结构，记录 prompt_tokens / completion_tokens / cached_tokens / latency_ms 等，每次调用后输出结构化日志。

---

### 问题 5：模型选择没有 Tiered Strategy

所有阶段用同一个模型。但不同阶段对模型能力的需求差异巨大。

**改造计划：** 新增 `StageModelConfig`，按阶段分配不同模型：

| Stage | 默认模型 | 理由 |
|-------|---------|------|
| summary | `deepseek-chat` | 压缩任务，轻量模型足够 |
| entities | `deepseek-chat` | 模式识别，结构化输出 |
| cards | `deepseek-v4-pro` | 核心质量环节，用主力模型 |
| graph | `deepseek-v4-pro` | 关系推理，需要理解力 |

如果用户只配置了一个模型，所有 stage 都用它（向后兼容）。

---

### 问题 6：JSON Mode 实现与 Provider 协议脱节

只实现了 OpenAI 的 `json_object`，但注册了 Anthropic/Gemini/Cohere 等协议。

**改造计划：** 按 `ProviderProtocol` 分支实现各自的 structured output 方式。

---

### 问题 7：缺乏细粒度 Stage 缓存

有 `CompileCache`（按 chunk 缓存完整结果），但没有 stage-level 缓存。

**改造计划：** 缓存 key = `SHA256(文档内容 + prompt 模板内容 + 模型名称)`。改 prompt 或换模型时自动失效。

---

### 问题 8：Max Tokens 配置静态且一刀切

`DOC_LIMITS` 是编译期常量，不会根据文档长度或模型能力调整。

**改造计划：** 根据 `model_info.max_output_tokens` 和文档长度动态计算。

## 三、问题 1 的讨论——Prompt Caching + 分类型调用

### 用户决策

- ✅ **保留分类型专用 prompt**（追求高质量卡片）
- ✅ **一次 input 建缓存，多次调用换 prompt**（解决重复 input 问题）
- ⏳ **是否实施：** 待代码改造后验证效果再决定

### 技术方案

```
第 1 次调用：文档内容 + 新知卡 prompt  → 创建 cache
第 2 次调用：cache_hit + 术语卡 prompt  → 复用 cache
第 3 次调用：cache_hit + 综述卡 prompt  → 复用 cache
第 4 次调用：cache_hit + 反常识卡 prompt → 复用 cache
```

**成本测算（15K tokens 文档，4 种卡片类型）：**

| 方式 | 第1次 | 第2-4次 | 总 input tokens |
|-----|-------|---------|----------------|
| 当前（无缓存）| 15K × 4 | 15K × 4 | **60K** |
| Prompt Caching | 15K × 1.25 (write) | ~1.5K (hit) × 3 | **23.25K** |

**边界条件：**
1. Cache 有效期 5 分钟（Anthropic）——当前流程 2 分钟内可完成，安全
2. 第一调用串行创建 cache，后续可并发引用
3. 不支持的 Provider 自动 fallback 到无缓存模式

### Provider 现状对缓存的影响

| Provider | 是否支持 Caching | 说明 |
|---------|-----------------|------|
| DeepSeek | ❌ 不支持 | 主要使用的 Provider，无缓存 API |
| LPGPT | ⚠️ 不确定 | 中转站，取决于上游 |
| ZSCC | ⚠️ 不确定 | 中转站，取决于上游 |
| Elysiver | ⚠️ 不确定 | 中转站，取决于上游 |
| Anthropic | ✅ 支持 | 有 API Key 但非主力 |

**结论：** 实现时必须做能力检测 + graceful fallback。DeepSeek 用户暂无法享受缓存收益，但代码结构需为未来预留。

## 四、用户澄清——卡片类型生成策略

### 原始设计 vs 用户意图

| 方面 | 代码现状 | 用户意图 |
|-----|---------|---------|
| 类型区分 | `required` / 非 `required` | **不存在这种区分** |
| 生成策略 | 只生成 `required == true` 的类型 | **所有类型都尝试生成** |
| 质量控制 | 无 | **生成后统一质量检测，不过关的说明原因** |

### 问题根因

`generate_cards()` 第 252 行：

```rust
for item in plan.iter().filter(|i| i.required) {
```

这行代码把 `CardPlanner` 中标记为 `required: false` 的类型全部过滤掉了。但这些类型（行动卡、人物卡、金句卡、事件卡、索引卡）是有实际 prompt 模板的，只是从未被调用。

### 修复方向

去掉 `required` 过滤逻辑，改为：

```rust
for item in plan.iter() {
    // 所有类型都尝试生成
    // LLM 根据文档内容决定是否有相关内容
    // 如果没有，返回空列表或"无相关内容"
}
```

然后统一进入质量检测流程，对空结果或低质量结果做标记说明。
