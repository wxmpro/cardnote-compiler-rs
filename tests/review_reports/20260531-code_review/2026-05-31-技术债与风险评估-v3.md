# cardnote-compiler-rs 技术债与风险评估 v3

> 评估日期：2026-05-31
> 评估范围：全量源码（~25,000 行 Rust + 16 个 Prompt 模板 + Python 脚本）
> 评估基准：v0.1.23（1f3f018），共 40 个 commit

---

## 🔴 高风险（可能导致生产事故）

### 1. `Box::leak` 内存泄漏 — 每次调用 `get_provider_config()` 都泄漏

**位置**：`src/config.rs:228-230`

```rust
provider: Box::leak(p.id.clone().into_boxed_str()),
base_url: Box::leak(p.default_base_url.clone().into_boxed_str()),
default_model: Box::leak(default_model.id.clone().into_boxed_str()),
```

**问题**：用 `Box::leak` 把 `String` 转为 `&'static str`。每次调用都永久泄漏 3 个 String。虽然 provider 数量有限不会直接 OOM，但属于明显的设计缺陷。

**根因**：`ProviderConfig` 结构体持有 `&'static str` 而非 `String`，导致必须使用 leak 手段。

**修复建议**：将 `ProviderConfig` 改为持有 `String` 字段，消除 `Box::leak`。

---

### 2. 字符串截断逻辑未标注省略号

**位置**：`src/main.rs:275-278`

```rust
let file_display = if file.chars().count() < 60 {
    file.clone()
} else {
    file.chars().rev().take(57).collect::<Vec<_>>().into_iter().rev().collect()
};
```

**问题**：截断后没有加 `...` 后缀，用户无法区分"文件名恰好 57 字符"和"文件名被截断"。低风险但容易误读。

---

### 3. Git 历史可能存在 API Key 泄露

**现状**：`.env` 文件当前仅包含一行注释，但 `.gitignore` 是后续添加的。需检查早期 commit 中是否提交过真实的 API Key。

**排查命令**：
```bash
git log --all -p -- .env .cardnote/providers.json | grep -i "sk-\|api_key\|token"
```

**修复建议**：若发现泄露，立即轮换相关 Key 并使用 `git filter-repo` 清理历史。

## 🟡 中等技术债（代码质量 / 可维护性）

### 4. converter.rs — 大量重复代码（3 个近乎相同的 MinerU 调用块）

**位置**：`src/converter.rs`

| 函数 | 行数范围 | 重复程度 |
|------|----------|----------|
| `read_pdf_scan()` | ~568-644 | MinerU 命令构建 + 输出解析 |
| `read_pdf_scan_fallback_mineru()` | ~495-566 | 几乎完全相同 |
| `read_pdf_expert_ocr()` 的 fallback 路径 | ~391-492 | 部分重复 |

**问题**：任何超时参数、标志位、输出格式的修改都要改 3 处，遗漏风险高。

**修复建议**：提取 `run_mineru(file_path: &str) -> Result<String>` 公共函数。

---

### 5. providers.rs 单文件 1231 行 — 巨型模块

**位置**：`src/providers.rs`

该文件承载了 4 类不同职责：
| 职责 | 行数占比 |
|------|----------|
| 14 个 Provider 硬编码定义 | ~77%（~950 行） |
| 凭据扫描逻辑 | ~10% |
| 配置文件解析 | ~8% |
| 状态报告生成 | ~5% |

**修复建议**：拆分为 `providers/registry.rs` + `providers/builtin.rs` + `providers/credentials.rs`。

---

### 6. 3 种协议是死代码 — 只有 OpenAI 兼容协议真正可用

**位置**：`src/api.rs:320-321`

`send_request()` 硬编码调用 `/chat/completions` endpoint。`ProviderProtocol` 定义了 4 种协议，但：
- `Anthropic` — `is_supported()` 返回 `false`，无法使用
- `Gemini` — `is_supported()` 返回 `false`，无法使用
- `Cohere` — `is_supported()` 返回 `false`，无法使用

注册表中 Anthropic/Gemini/Cohere 的 Provider 条目及其 `ModelInfo` 均为死代码。

**修复建议**：要么实现多协议 dispatch，要么删除不可达的 Provider 注册。

### 7. 测试覆盖：零集成测试

**现状统计**：
| 测试类型 | 数量 |
|----------|------|
| 单元测试（`#[test]`） | 195 个 |
| 集成测试 | 0 个 |
| 端到端测试 | 0 个 |

**核心盲区**：
- 从文档输入到卡片输出的完整流水线从未被端到端验证
- LLM 响应解析逻辑只在单元测试中用手工 JSON 验证，未测试真实 LLM 输出
- Map-Reduce 分块 → 合并 → 去重的完整路径无测试

**修复建议**：至少添加 1 个基于 fixture 文档的 `pipeline.run()` 集成测试（mock LLM 响应）。

---

### 8. 依赖外部 Python 工具链未做版本约束

**位置**：`src/converter.rs`

调用了 4 个外部 Python 工具，但无版本检查、无能力探测：

| 工具 | 调用方式 | 风险 |
|------|----------|------|
| `pymupdf` | `python3 -c "import pymupdf..."` | API 变化无感知 |
| `markitdown` | `python3 -c "from markitdown..."` | 同上 |
| `pytesseract` | `python3 -c "import pytesseract..."` | 系统可能无 tesseract 二进制 |
| `mineru` | `Command::new(&mineru_path)` | 版本/参数兼容性未知 |

**额外风险**：`Command::new("python3")` 在某些系统上可能指向 Python 2 或不存在。

**修复建议**：启动时运行 `doctor` 检测 Python 版本 + 各工具可用性，结果缓存。

## 🟠 架构风险（随规模增长会恶化）

### 9. `ProviderRegistry::new()` 被频繁调用 — 每次重建整个注册表

**调用点**：
- `get_provider_config()` — 每次编译
- `create_client()` — 每次编译
- `get_provider_label()` — 每次编译
- `scan_credentials()` — 每次编译
- `interactive_setup()` — 配置向导

每次调用涉及 HashMap 构建 + 可能的文件 I/O（读 `providers.default.json`）。

**修复建议**：改为 `static REGISTRY: LazyLock<ProviderRegistry>`。

---

### 10. 缓存清理在 `Pipeline::new()` 中同步阻塞

**位置**：`src/pipeline.rs:404`

```rust
// 启动时清理过期缓存
cleanup_cache_dir();
```

`cleanup_cache_dir()` 阻塞主线程扫描文件系统。当 `.cardc_cache/` 中有大量文件时，会明显延迟第一次编译的启动时间。

**修复建议**：异步化（`tokio::spawn`）或移至后台线程。

---

### 11. Map-Reduce 并发数硬编码为 2

**位置**：`src/pipeline.rs:667`

```rust
let semaphore = Arc::new(Semaphore::new(2));
```

对于大文档（20+ 块），2 并发的吞吐量受限。应可通过 CLI 参数或环境变量配置。

---

### 12. LLM 响应无 JSON Schema 验证 — 空对象 `{}` 会静默通过

**位置**：`src/api.rs:210-216`

`chat_json()` 只要求返回合法 JSON，不验证是否包含预期字段。LLM 返回空对象 `{}` 或字段名不匹配时：
- 摘要阶段 → 空 `Summary { title: "", overview: "", ... }`
- 卡片阶段 → 空 `Vec<Card>`
- 图谱阶段 → 空 `KnowledgeGraph { entities: [], relations: [] }`

这些"空结果"会静默进入输出，用户可能误以为文档确实没有内容。

**修复建议**：每个阶段的 JSON 解析后添加最少字段校验（非空 title / 非空 cards 列表等）。

## 从 Git 历史看演进模式

23 个版本中的高频修复主题：

| 主题 | 涉及版本数 | 说明 |
|------|-----------|------|
| ref 格式 / 卡片解析 | v0.1.7 → v0.1.15（5 个版本） | LLM 输出格式不稳定 |
| PDF 转换 / OCR fallback | v0.1.3 → v0.1.12（4 个版本） | 外部工具链脆弱 |
| 质量过滤误杀 | v0.1.6 → v0.1.7（2 个版本） | 启发式规则边界多 |

**核心洞察**：LLM 输出解析是整个系统最脆弱的环节。没有 schema 验证 + 没有 structured output = 每次 LLM provider 升级模型都可能引发格式回归。v0.1.15-v0.1.18 连续 4 个 patch 都在修 ref 格式和 JSON 截断，验证了这一点。
