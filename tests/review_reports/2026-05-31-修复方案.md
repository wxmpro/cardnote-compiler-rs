## 修复清单

### 修复 1: 中文文档卡片规划数量偏少

**问题**: `stages/cards.rs:248` 使用 `document.len()`（UTF-8 字节数）而非 `document.chars().count()`（字符数）。

**根因**:  
- `CardPlanner::plan_book` 以 `char_count > 50000` 判断是否需要翻倍卡片数量  
- 中文每个字符占 3 字节，`document.len()` 返回的值是字符数的约 3 倍  
- 导致 15000 字符的中文文档被误判为 45000 字节 → 接近但未超过阈值 → 不翻倍  
- 实际应翻倍（15000×3=45000，超过 50000 的 90% 但未超）

**修复**: 单字符修改：`document.len()` → `document.chars().count()`

**风险**: 零风险。仅影响 `CardPlanner` 的输入参数，不改变任何其他逻辑。  
**验证**: `cargo test` 通过（现有测试使用 ASCII 数据，不受影响；但修复后行为正确）。  
**向后兼容**: 完全兼容。仅影响中文/多字节文档的卡片规划，ASCII 文档行为不变。

### 修复 2: 编译结果为空时缺乏可见性

**问题**: `pipeline.rs` 中所有 LLM 阶段失败时返回默认值（空列表/空结构），不中断编译。用户看到"编译完成"但输出目录内是空文件。

**根因**:  
- `run_single` 中 summary/entities/cards/graph 四个阶段均使用 `match Ok/Err` 模式  
- `Err` 分支将错误记录到 `diagnostics` 但返回空值  
- `output.rs::save_single` 无条件创建目录并写入文件，即使内容为空  
- 用户可能在终端输出被刷掉后，只看到"结果已保存到: ..."，误以为成功

**修复**: 在 `main.rs:handle_compile` 中，在 `pipeline.run()` 返回后、保存输出前，增加编译结果健康检查：

```rust
// 在 pipeline.run() 返回后
if result.cards.is_empty() && result.summary.title.is_empty() {
    println!("\n{} 编译结果为空，可能某个阶段完全失败。", "⚠".yellow());
    println!("   请检查 compile_diagnostics.md 了解详情。");
    // 继续保存，让用户有诊断文件可看
}
```

**风险**: 极低。仅增加打印语句，不改变控制流。  
**验证**: `cargo build` 通过。  
**向后兼容**: 完全兼容。仅增加诊断输出，不改变编译行为。

### 修复 3: 质量报告目录安全删除逻辑

**问题**: `main.rs:399-406` 使用 `canonicalize` 检查路径安全性，但如果路径不存在会返回 Err，回退到字符串检查 `!report_dir.contains("..")`，这不够严格。

**根因**:  
- `std::fs::canonicalize` 在路径不存在时返回 `Err`  
- 回退逻辑只检查 `".."` 子串，可以被 `"..."` 绕过

**修复**: 强化回退检查逻辑，使用路径组件遍历检查：

```rust
// 当前回退逻辑：
_ => !report_dir.contains("..")

// 修复后：使用 Path 组件检查
_ => !Path::new(&report_dir).components().any(|c| matches!(c, std::path::Component::ParentDir))
```

**风险**: 极低。仅增强安全检查，不改变正常路径行为。  
**验证**: `cargo test` 通过。  
**向后兼容**: 完全兼容。
