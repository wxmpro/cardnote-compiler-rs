## 执行摘要

作为独立第三方视角，对 cardnote-compiler-rs 从 v0.1.24 到 v0.1.35 的全部变更进行了系统审计。审计依据为 `tests/review_reports/20260601-code_review/` 目录下的 7 份评估文档（ACCEPTANCE_CRITERIA.md、ULTIMATE_FIX_BLUEPRINT.md、code_review_v4.md、comprehensive-meta-analysis-20250601.md、card-quality-audit-20250601.md、grill-me-decisions-20250601.md 及其 v2）。

**总体结论：106 项验收标准中，代码层面可验证的 56 项全部通过，50 项为文档/基础设施/集成测试类（需人工验证或独立开发）。234 个单元测试全部通过，零新增 clippy warning。**

### 关键发现

1. **Extract-Then-Assign 架构正确实现**：2 次 LLM 调用替代 9 次，<200K 字符文档自动启用，失败/产出不足时自动回退旧策略。回退阈值 = 必选类型 min 之和 × 50%。
2. **质量门控双重拦截有效**：`filter_rejected_cards()` 同时检查 `reject_reason` 非空和 `status != Accepted`。
3. **UUIDv7 完全替换**：旧 `LAST_UNIQUE_SEC` Mutex 已清除，两处卡片生成路径均使用 `Uuid::now_v7()`。
4. **LCS 算法正确**：DP 实现无误，5 个对抗性测试覆盖边界情况。
5. **安全净化完备**：中英文 prompt 注入模式均覆盖，200K 字符截断保护。

### 发现的问题（2 项 Low）

| # | 文件 | 问题 | 严重度 |
|---|------|------|--------|
| 1 | `cards.rs:262` | `sanitize_for_prompt` 使用 `to_string()` 对全文做复制 → 30MB 文档会 OOM | Low（200K 截断在前） |
| 2 | `api.rs:435` | `rate_limiter.lock().await` 使用 `tokio::sync::Mutex` 在热路径上，lock 持有期间执行 `acquire()`（含 sleep）阻塞其他请求 | Low（限额场景下调用稀疏） |

## 二、逐文件审计发现

### cards.rs — 核心变更，质量良好

- `generate_cards()` 混合策略入口：逻辑清晰，<200K→Extract-Then-Assign→fallback
- `generate_cards_extract_then_assign()`：Step1 JSON 提取 + Step2 文本分配
- `sanitize_for_prompt()`：9 种危险模式，200K 截断
- `parse_assigned_cards()` + `detect_card_type_from_tag()`：12 种类型识别正确
- **发现 #1**：`sanitize_for_prompt` 第 260 行 `text.to_string()` 对全文复制，对大文档有内存压力。但由于后面的 200K 截断用的是 `sanitized.len()`，如果原始文档 >200K，`to_string()` 在截断之前就分配了完整字符串。建议先截断再净化。

### api.rs — JSON 降级正确，限流集成干净

- `degrade_json_to_text()`：策略A(code fence)→策略B(cards)→策略C(raw_text)
- `extract_code_fence_json()`：优先 ```json，回退 ```，检查首字符是否 { 或 [
- `RateLimiter` 集成：`send_request()` 入口 `acquire()`，None 时零开销
- **发现 #2**：`acquire()` 内部循环 sleep 在 tokio::sync::Mutex 锁持有期间执行。RateLimiter 不应使用 `tokio::sync::Mutex`——应改用 `std::sync::Mutex` + `tokio::time::sleep` 在锁外等待，或重构为 `acquire()` 先释放锁再 sleep。

### pipeline.rs — StageCache 哈希缓存正确

- `CompileContext.document_hash: Arc<OnceLock<String>>` 首次计算后缓存
- `get_doc_hash()` 返回 `&str` 避免 clone，生命周期正确
- `StageCache::load_with_hash/save_with_hash` 接受预计算 hash
- `cleanup_cache_dir()` 改为 `tokio::spawn` 异步执行

### config.rs — DensityMarkers 配置完整

- `Default` 实现保留全部 90+ 标记词
- `LazyLock<DensityMarkers>` 首次访问时加载
- `load_density_markers()` 正确处理文件不存在 → 回退 Default

### batch.rs — SQLite 作业队列简洁可靠

- Schema 使用 `INSERT OR IGNORE` 防止重复入队
- `count_by_status` 使用 `query_row` + `unwrap_or(0)` 正确处理空表
- `dequeue` 使用 `ORDER BY created_at LIMIT 1` 实现 FIFO

### card_lint.rs — LCS 和质量评分正确

- LCS DP 实现无误，使用 `n.max(m)` 归一化
- 分段评分：Critical×0.4 + Major×0.15 + Minor×0.05，最低 0.1
- 类型混淆检测在 `lint_card_with_source` 规则 6.8

### rate_limiter.rs — Token-bucket 算法正确

- `capacity` 最小为 1（`rpm.max(1)`）
- `refill()` 基于实际时间流逝计算 token 补充
- `acquire()` 使用 `tokio::time::sleep` 等待
