# CardNote Compiler — 生产就绪评估

> 评估日期：2026-06-02
> 评估版本：v0.1.38
> 评估方法：全管线逐环节极端情况推演

---

## 一、全管线极端情况矩阵

### 环节 1：CLI 入口

| 极端情况 | 是否处理 | 处理方式 |
|---------|---------|---------|
| 文件路径含 null 字节 | ✅ | `converter.rs:56` 拒绝 |
| 文件不存在 | ✅ | `converter.rs:60` 返回 `FileNotFound` |
| 无参数运行 | ✅ | 提示 `cardc init/doctor` |
| 大文件(>500MB) | ✅ | `MAX_FILE_SIZE_MB=500` 拒绝 |
| 大文本(>50MB) | ✅ | 流式读取，防 OOM |
| 损坏的 PDF | ⚠️ | lopdf 解析失败 → PyMuPDF fallback → 可能返回空文本 |
| 加密 PDF | ❌ | **未处理** — 需要密码的 PDF 会直接失败，错误信息不友好 |

### 环节 2：环境

| 极端情况 | 是否处理 | 处理方式 |
|---------|---------|---------|
| Python 未安装 | ❌ | `converter.rs` 调用 `python3` 失败时返回通用错误 |
| Python 包缺失 | ❌ | **未做版本检查** — pdymupdf/markitdown 缺失时运行时 crash |
| Rust 版本不匹配 | ✅ | `cargo build` 时自动检测 |
| macOS 非 14+ | ❌ | **未检查** — lopdf 等依赖可能不兼容旧版 macOS |
| 磁盘空间不足 | ❌ | **未检查** — 写输出时 `fs::write` 失败，错误信息是底层 IO error |

### 环节 3：提供商

| 极端情况 | 是否处理 | 处理方式 |
|---------|---------|---------|
| 无 API key | ✅ | 交互式配置向导 |
| API key 无效(401) | ✅ | `send_request` 返回 `AppError::Api` |
| API 限流(429) | ✅ | `CARDNOTE_MAX_RPM` 限流器 + 3 次重试 |
| API 超时 | ✅ | 120s HTTP timeout + `with_retry` 3 次重试 |
| 模型不支持 JSON mode | ✅ | 自动切换到 fallback 模型 |
| 提供商全部不可用 | ✅ | `health::check_all_providers` 预检 |
| API 返回非标准格式 | ⚠️ | `extract_content` 只处理 OpenAI/Anthropic 两种格式 |
| API 返回超大响应 | ❌ | **无响应大小限制** — 可能 OOM |
| 网络断开 | ⚠️ | reqwest 超时后报错，但无网络恢复后的自动重连 |
| LLM 返回有害内容 | ❌ | **无输出内容审核** — prompt 注入净化只做了输入侧 |

### 环节 4：PDF 扫描

| 极端情况 | 是否处理 | 处理方式 |
|---------|---------|---------|
| 文字层 PDF | ✅ | `lopdf` 检测 `Resources→Font` |
| 扫描版 PDF | ✅ | 提示需 OCR，`--force` 可跳过 |
| 手绘/矢量 PDF | ✅ | 归类为 HandDrawn |
| 空 PDF(0 页) | ⚠️ | lopdf 解析失败 → Error 状态 |
| 混合 PDF(部分页扫描) | ❌ | **只采样前 5 页** — 第 6 页起是扫描版无法检测 |
| 超大 PDF(>1000 页) | ⚠️ | 采样 5 页 + TOC 分页，但 5000 页会超时 |

### 环节 5：转换

| 极端情况 | 是否处理 | 处理方式 |
|---------|---------|---------|
| MarkItDown 失败 | ✅ | 三层 fallback: MarkItDown → PyMuPDF → MinerU |
| PyMuPDF 超时 | ✅ | `CONVERT_TIMEOUT_SECS=300` |
| OCR 质量极差 | ⚠️ | 生成低质量文本 → 质量门控可能拦截 |
| 非 UTF-8 编码 | ✅ | 自动检测 UTF-8/16/Shift_JIS/EUC-KR/GBK/BIG5 |
| 纯图片 PDF(无任何文字) | ⚠️ | OCR 路径，但可能完全失败 |
| 页眉页脚污染 | ✅ | `pdf_postprocess.rs` 检测并移除重复 header/footer |
| 转换超时(1800s OCR) | ✅ | `OCR_TIMEOUT_SECS=1800` |

### 环节 6：质量门

| 极端情况 | 是否处理 | 处理方式 |
|---------|---------|---------|
| 转换后文本为空 | ⚠️ | 质量评分为 F → `is_acceptable()=false` |
| 转换后全是乱码 | ✅ | `compute_garbage_ratio` 检测控制字符/替换字符 |
| 正常文档但质量评分低 | ✅ | `--force` 可跳过门控 |
| 评分边界值(恰好 D/F) | ✅ | 明确阈值 |

### 环节 7：编译（核心）

| 极端情况 | 是否处理 | 处理方式 |
|---------|---------|---------|
| 文档 > 200K 字符 | ✅ | Extract-Then-Assign 跳过，直接用 legacy |
| Extract-Then-Assign 产出不足 | ✅ | `< min_cards_threshold` → fallback |
| Extract-Then-Assign 完全失败 | ✅ | `generate_cards_legacy` 兜底 |
| JSON 解析全部重试失败 | ✅ | `degrade_json_to_text` 文本降级 |
| 某类型 prompt 缺失 | ✅ | `load_prompt("all_cards")` fallback |
| 某类型卡片完全不生成 | ❌ | **无类型覆盖检查** — 只查总数不满阈值才 fallback |
| Map-Reduce 0 chunk | ✅ | v0.1.38 已加防护 |
| Map-Reduce 全部分块失败 | ✅ | 失败率 >20% 中断 |
| 卡片内容含 Markdown 破坏字符 | ✅ | `escape_markdown()` |
| LLM 返回非预期卡片类型 | ❌ | `detect_card_type_from_tag` 只能识别特定标签，归为 Knowledge |
| LLM 完全不遵循 prompt 格式 | ⚠️ | `parse_single_type_cards` 按 `---` 分割 + `extract_field`，格式若完全不同则卡片为 0 |

### 环节 8：输出

| 极端情况 | 是否处理 | 处理方式 |
|---------|---------|---------|
| 0 张卡片通过门控 | ✅ | 输出空 `all_cards.md` + 质量报告记录被拦截原因 |
| 输出目录已存在 | ✅ | `resolve_unique_output_dir` 自动追加 `_001` |
| 磁盘写满 | ❌ | fs::write 失败，错误信息是底层 IO error |
| 文件名含非法字符 | ✅ | `sanitize_filename()` |
| 并发写 SQLite | ✅ | v0.1.38 已加 WAL 模式 |

## 二、评分

| 维度 | 评分 | 说明 |
|------|------|------|
| **管线完整性** | 7/10 | 大部分极端情况有处理，但遗漏了加密 PDF、Python 依赖检查、响应大小限制 |
| **错误恢复** | 7/10 | 多层 fallback 设计良好，但 0 卡片产出的诊断信息不够详细 |
| **安全** | 6/10 | Prompt 注入已处理，但无输出内容审核、磁盘满和网络断开的处理粗糙 |
| **运维** | 5/10 | 无监控指标、无健康检查 API、无结构化日志、无 alert |
| **用户体验** | 6/10 | CLI 友好，但错误信息有时是底层 IO/API 错误，用户难以理解 |
| **文本生成类特有** | 6/10 | 类型覆盖缺失是最大盲区，LLM 输出审核完全空白 |

**综合：6.5/10**

## 三、能否作为生产级项目上线？

**答：不能。**

差距不在于"会不会 crash"——管线本身有良好的 fallback 设计。差距在于**运维边界**：

1. **LLM 输出不可审计**：没有类型覆盖检查、没有内容审核。5000 本书可能有数百本产出 0 张反常识卡而无人知晓
2. **故障不可观测**：无结构化日志、无 Metrics、无 Alert。用户不知道 Extract-Then-Assign 回退率是多少
3. **边界不够硬**：磁盘满、加密 PDF、Python 缺失这些常见用户场景的体验很差
4. **LLM 特有风险未覆盖**：cost runaway（无费用上限）、prompt drift（模型升级后行为变化）、output contamination（有害内容）

## 四、能获得 10000 star 吗？

**答：需要先补齐三类差距。**

10000 star 项目的共同特征：
- 安装一行命令（✅ cargo build --release）
- 使用一行命令（✅ cardc book.pdf）
- 效果惊艳的 GIF 演示（❌ 没有）
- 活跃的社区 + issue 响应（❌ 未开始）
- 清晰的文档 + 对比示例（⚠️ README 够用但缺 before/after 截图）
- 解决一个很多人痛的问题（✅ PDF→知识卡片的刚需）
- 与主流生态集成（❌ 无 Obsidian/Logseq/Anki 插件）

**最快路径**：
1. 补类型覆盖检查 + 基础监控日志（1 天）
2. 做 10 本书的 before/after 示例 + GIF 演示（半天）
3. 发布到 Rust 中文社区 + V2EX + 即刻/小红书（半天）
4. Obsidian 插件（1 周）

在生态系统就绪前，star 数天花板约 200-500。
