## 一、元（Meta）—— 对"修复"本身的审视

### 1.1 根因追溯：修的是症状还是根因？

| 原始问题（54项） | 修复方式 | 根因判断 |
|-----------------|---------|---------|
| ref 格式违规（12项同源） | Prompt 约束 + `check_ref_format` + `fix_ref_format` 10条规则 | **症状治了大半，根因未完全消除**。Prompt 能减少但无法杜绝 LLM 格式偏差，`fix_ref_format` 的 10 条规则本质是"补丁叠补丁"——v0.1.15 加了规则、v0.1.17 提取了 KNOWN_BOOKS、v0.1.35 配置化 books.json。但规则 8/9/10 在 fallback 时仍然硬编码 "人生模式" |
| 复制检测算法错误 | 字符包含率→LCS | ✅ **根因修复**。算法从数学层面纠正 |
| 唯一ID冲突 | Mutex+时间戳→UUIDv7 | ✅ **根因修复**。消除了三个隐含假设 |
| 9次LLM调用 | Extract-Then-Assign | ⚠️ **根因部分修复**。调用次数降了，但"每次传输完整文档"的问题仍在——提取步骤仍传全文，只是从9次变成了1次。真正根因是"文档与prompt的耦合方式" |

**关键洞察**：`fix_ref_format` 的演进路径（分散硬编码→KNOWN_BOOKS常量→books.json配置）是正确的根因修复方向——从"修补表面"到"改变结构"。但 Extract-Then-Assign 并没有改变"把整个文档塞进prompt"这个根本模式，只是减少了次数。

### 1.2 过度设计审查

| 变更 | 过度设计？ | 判断理由 |
|------|----------|---------|
| SQLite 批处理队列 | **否** | Craftsman 模式不需要批量编排，但 `cardc batch` 是用户明确需求（5000 本 PDF）。SQLite 选型合理——零外部依赖、嵌入式、WAL 模式支持并发读 |
| 密度标记词 TOML 配置 | **边缘** | 90+ 标记词的配置化在日常使用中几乎不会被修改。但 `Default` 实现保留了全部默认值，opt-in 设计无额外负担 |
| RPM 限流器 | **否** | 5000 本 × 多次调用 = 数万次 API 请求。无限流保护等于裸奔 |
| JSON 三策略降级 | **否** | 策略A/B/C 从严格到宽松逐层降级，每一层都有明确的触发条件 |

### 1.3 版本管理策略

**现有模式**：`v0.1.XX` 线性递增，每次推送一个版本。

**问题**：本次变更从 v0.1.24 → v0.1.35，跨度 11 个版本号。其中 v0.1.26-30 是 grill-me 文档的反复 revert/重写。这不是版本管理问题，而是**工作流问题**——grill-me 交互应该在一个分支内完成后再合并，而不是每次输出都提交到 main。

**建议**：采用 `feature/` 分支开发 → squash merge → main 单版本发布的模式。本次的 feature 分支实践已经验证了这个模式可行。

### 1.4 测试策略审视

234 个测试中，**0 个是集成测试**。核心路径（Extract-Then-Assign、JSON 降级、Batch CLI）完全依赖人工验证。这不是"测试不足"的问题——对于 CLI 工具，mock LLM 响应的难度确实高。但至少应该：

- 为 `sanitize_for_prompt` 添加单元测试（纯函数，易测试）

- 为 `parse_assigned_cards` 添加 mock 响应测试

## 二、反（Anti）—— 反向思维与边界检验

### 2.1 Extract-Then-Assign 可能比 9 次调用质量更差

**对立面分析**：

| 场景 | 9次调用（旧） | 2次调用（新） | 谁更好？ |
|------|-------------|-------------|---------|
| 短书（<5万字），概念密集 | 每种类型独立 prompt，LLM 专注于单一认知动作 | 一次提取所有知识点，可能遗漏"术语的微妙定义"与"反常识的认知张力" | **旧策略可能更好** |
| 长书（10-20万字），概念稀疏 | 9次调用大量重复传文档，LLM 注意力稀释 | 提取+分配，避免重复传输 | **新策略明显更好** |
| 小说/文学作品 | 9次调用可能生成 0 张术语卡（合理）但浪费调用 | 提取步骤可能强行给小说生成伪术语卡 | **旧策略更诚实** |
| 技术手册 | 术语卡 3×scale 张，覆盖充分 | 提取步骤可能遗漏不常见但重要的术语 | **需实测对比** |

**关键发现**：回退阈值 `min_cards_threshold = 必选min之和 ÷ 2` 过于粗糙。对于 Book 类型 scale=1 时，必选 min 之和 = 3+2+1+1 = 7，阈值 = 3。新策略只要生成 3 张卡片就"通过"——此时 9 次调用可能生成了 15 张。**阈值应该基于实际规划总数而非必选 min 之和的一半**。

**改进建议**：
```rust
fn min_cards_threshold(doc_type: DocumentType, char_count: usize) -> usize {
    let plan = CardPlanner::plan(doc_type, char_count);
    let total_min: usize = plan.iter().map(|p| p.min).sum();
    (total_min as f64 * 0.3) as usize // 改为总 min 的 30%
}
```
这从 7→3 变为 16→5（scale=1），更重要的是覆盖了可选类型。

### 2.2 Batch CLI 的 SQLite 是否过度？

**对立面**：一个简单的 JSON 文件（`processed.json`）就能跟踪 5000 个文件的状态——为什么需要 SQLite？

**辩护**：
- JSON 文件的并发写入需要文件锁，SQLite 自带 WAL 并发控制
- SQLite 的 `INSERT OR IGNORE` 天然去重
- 查询性能（`SELECT COUNT WHERE status='pending'`）远远优于遍历 JSON
- SQLite 的 `bundled` feature 编译进二进制，零外部依赖

**结论**：SQLite 不是过度，是正确选择。

### 2.3 限流器在什么场景下反而有害？

**反向场景**：
- `CARDNOTE_MAX_RPM=5` + 每本书 2 次调用 = 每本书等待 ~24 秒（两次调用间隔 12 秒）
- 5000 本书 = 额外等待 ~33 小时
- 如果用户不知道限流器已启用，会认为程序"卡住了"

**建议**：限流器等待时打印提示（"⏳ RPM 限流，等待 X 秒..."）。

### 2.4 质量门控拦截是否可能误杀高质量卡片？

**当前逻辑**：`filter_rejected_cards` 拦截所有 `reject_reason` 非空或 `status != Accepted` 的卡片。

**误杀风险**：
- 卡片标题与内容不匹配（`TitleContentMismatch`）→ 可能是 LLM 用了不同措辞但语义相同
- 金句卡缺少出处（`QuoteMissingSource`）→ 可能原文确实没有明确出处

**实际的兜底**：被拦截的卡片仍写入 `card_quality_report.md`，用户可以在人工审阅环节恢复。所以不是永久丢失——这个设计是对的。

## 三、空（Void）—— 解构性质疑

### 3.1 当前架构的根本假设

```
假设 1: LLM 能准确理解 prompt 中的类型定义
  └── 如果错误 → 卡片类型错配（但 parse_assigned_cards 从 #标签 纠正）

假设 2: 文档可以被完整塞进 LLM 上下文窗口
  └── 如果错误 → 200K 字符保护 + Map-Reduce fallback

假设 3: 提取步骤产生的知识点的 JSON，能被分配步骤的 LLM 正确处理
  └── 如果错误 → 分配步骤会生成格式错误的卡片（parse_assigned_cards 容错）

假设 4: 每本书独立处理，不需要跨书知识
  └── 这是用户明确确认的边界（Craftsman 模式）

假设 5: 用户会人工审阅卡片产出
  └── 质量门控拦截的卡片在 card_quality_report.md 中可恢复
```

### 3.2 Extract-Then-Assign 的隐含前提

**前提**：提取出的 JSON 知识点列表，包含了原文档的**所有有价值信息**，且**足够详细**让分配步骤能准确判断类型。

**如果前提不成立**：
- 提取步骤遗漏了某个概念的微妙定义 → 分配步骤永远无法生成对应的术语卡 → 该类型卡片永久丢失（除非 fallback 触发）
- 提取步骤把两个相关概念合并 → 分配步骤可能生成一张"杂糅卡"而非两张独立卡
- 提取步骤的知识点缺乏上下文章节信息 → 分配步骤无法判断是"新知"还是"反常识"

**实际缓解**：`min_cards_threshold` 提供了一道安全网。但如上所述（反 2.1），这个阈值设置可能过于宽松。

### 3.3 尚未覆盖的盲区

| 盲区 | 描述 | 严重度 |
|------|------|--------|
| **B1: 提取步骤的知识点质量无验证** | `generate_cards_extract_then_assign` 对提取的 JSON 只检查 `knowledge_points` 是否非空——不检查每个知识点的 title/content/evidence/ref 是否完整。如果 LLM 返回了 50 个知识点但 30 个没有 evidence，没有检测。 | 中 |
| **B2: 非 UTF-8 文档的 truncate 不安全** | `sanitize_for_prompt` 使用 `truncate(max_len)` 按字节截断，可能截断在多字节 UTF-8 字符中间。应用 `char_indices` 安全截断。 | 低（200K 边界极少触发） |
| **B3: Batch CLI 的 error 字段可能包含 API key** | `mark_failed` 将完整的错误信息存入 SQLite 的 `error` 字段。如果 API 调用返回 401 且错误消息中包含 key 前缀，key 会被写入 `.cardnote/batch.db`。 | 中（安全敏感） |
| **B4: RateLimiter 无最大等待时间** | `acquire()` 的循环是无限的。如果 `CARDNOTE_MAX_RPM=0`（被误设为 0 → capacity=1，rate=1/60 RPS），每次调用等待 60 秒但没有超时退出。 | 低（用户不会设 RPM=0） |
| **B5: 分配步骤的 LLM 可能"偷懒"** | `assign_cards.md` 给 LLM 输入了所有知识点和完整的卡片类型定义。LLM 可能选择只生成 3-4 种简单类型（术语卡、基础卡）而跳过需要更多认知动作的类型（反常识卡、新知卡）。 | 高 |

**B5 是最危险的盲区**。分配步骤的 prompt 中没有"禁止只生成简单类型"的约束——LLM 天然倾向于省力。如果 LLM 发现把知识点全写成术语卡比写成反常识卡更省事，它就会这么做。而 `min_cards_threshold` 只看数量，不看类型覆盖。

**建议修复**：在 `assign_cards.md` 的"分配策略"中增加：
```
## 硬性约束
每种必选类型至少生成 min 张卡片。如果某个必选类型的卡片数量不足规划要求，
必须从现有知识点中选择最接近该类型的知识点重新分配。禁止将所有知识点全部分配为同一种类型。
```

### 3.4 概念纯洁性质疑

**"Craftsman-Grade Production" 是否是一个矛盾的术语？**

Craftsman（工匠）暗示单件精作。Production（生产）暗示批量复现。当前架构试图在两者间找平衡——但不应该期望一个工具同时满足：

- 单本精作：需要高质量的类型特定 prompt、人工审阅、卡片迭代
- 5000 本规模：需要批量处理、自动化、容错

**建议**：v0.2.0 应该明确声明工具的定位——是"批量预处理工具"（快速过 5000 本，生成初稿卡片，人工精选后精修），还是"单本精作助手"（慢但质量高）。两者需要不同的优化方向。
