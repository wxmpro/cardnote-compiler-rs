## 一、项目架构概述

### 1.1 核心编译流水线（4 阶段）

```
文档输入 → [转换器] → Markdown 文本
              ↓
         [文档类型检测] → 确定卡片规划策略
              ↓
    ┌─────────────────────────────────────┐
    │  单轮编译（< 50K 字符）              │
    │  或 Map-Reduce 分块编译（≥ 50K 字符）│
    └─────────────────────────────────────┘
              ↓
    ┌─────────┬─────────┬─────────┬─────────┐
    │ Stage 1 │ Stage 2 │ Stage 3 │ Stage 4 │
    │ 摘要    │ 实体    │ 卡片    │ 图谱    │
    │ summary │ entities│ cards   │ graph   │
    └─────────┴─────────┴─────────┴─────────┘
              ↓
         [去重/合并/质量过滤]
              ↓
         Markdown 输出文件
```

### 1.2 卡片生成架构（关键路径）

```
all_cards.md (聚合 Prompt)
    ↓
CardPlanner::plan_text(doc_type, len) → 生成规划文本
    ↓
Prompt 模板替换 ({card_plan} + {document})
    ↓
单次 LLM 调用 (call_chat, 文本模式, 8000 tokens 限制)
    ↓
parse_all_cards() → 按 "---" 分隔解析卡片块
    ↓
extract_field(block, "类型") → 识别 CardType
    ↓
匹配失败 → 回退为 CardType::Knowledge (新知卡)
```

### 1.3 CardType 枚举（12 种）

| 枚举变体 | Prompt 名称 | 解析器匹配 | 规划中是否出现 |
|---------|------------|-----------|--------------|
| `Knowledge` | 新知卡 | "新知卡"\|"新知" | 是（所有类型） |
| `Term` | 术语卡 | "术语卡"\|"术语" | 是 |
| `Person` | 人物卡 | "人物卡"\|"人物" | 是（Book/Paper） |
| `Action` | 行动卡 | "行动卡"\|"行动" | 是 |
| `Quote` | 金句卡 | "金句卡"\|"金句" | 是 |
| `CounterIntuit` | 反常识卡 | "反常识卡"\|"反常识" | 是 |
| `Event` | 事件卡 | "事件卡"\|"事件" | 是（Book/Paper） |
| `Graph` | 图示卡 | "图示卡"\|"图示" | 是（Manual/Report） |
| `NewWord` | 新词卡 | "生字卡"\|"生字"\|"词汇卡" | **否** |
| `Note` | 基础卡 | "笔记卡"\|"笔记" | 是 |
| `Index` | 索引卡 | "目录卡"\|"目录"\|"索引卡" | 是 |
| `Review` | 综述卡 | "综述卡"\|"综述"\|"总结卡" | 是 |

**关键发现**：`NewWord`（新词卡）在 `CardPlanner` 的所有规划函数中完全缺失（`plan_book`/`plan_paper`/`plan_manual`/`plan_report`/`plan_article`）。枚举中存在但规划中被遗漏。

## 二、核心问题诊断：为什么只生成一种卡片类型

### 根因 1：Prompt 格式与解析器格式不匹配（**最高优先级**）

**问题描述**：`all_cards.md` 要求的输出格式与 `parse_all_cards` 期望的解析格式存在严重不匹配。

`all_cards.md` 要求的格式（v1 遗留）：
```
---
#卡片类型
标题：[卡片标题]
内容：[卡片正文内容]
ref：[当前文档] | [章节名] | 第[页码]页
uuid：[YYYYMMDDHHMM 格式]
---
```

`parse_all_cards` 期望的格式：
```rust
// cards.rs:306
let card_type = parse_card_type(block);  // 查找 "类型：" 或 "类型:"

// cards.rs:351-368
fn parse_card_type(block: &str) -> CardType {
    let type_str = extract_field(block, "类型").unwrap_or_default();
    match type_str.as_str() {
        "人物卡" | "人物" => CardType::Person,
        "术语卡" | "术语" => CardType::Term,
        // ...
        _ => CardType::Knowledge,  // ← 回退默认值
    }
}
```

**冲突点**：
- Prompt 要求 `#卡片类型`（Markdown 标题语法）
- 解析器查找 `类型：卡片类型`（键值对语法）
- 如果 LLM 严格遵循 Prompt 输出 `#新知卡`，解析器的 `extract_field(block, "类型")` 返回 `None`
- 所有卡片被回退为 `CardType::Knowledge`

**验证方法**：检查实际生成的 Markdown 输出文件，观察卡片是否以 `#新知卡` 开头而非 `类型：新知卡`。

### 根因 2：CardType 回退默认值硬编码（**高优先级**）

`cards.rs:366`：
```rust
_ => CardType::Knowledge,  // 所有无法识别的类型都回退为新知卡
```

这意味着：
- 即使 LLM 生成了多种类型的卡片
- 只要格式不被解析器识别，全部被统一为新知卡
- 用户看到的"只有一种卡片类型"实际上是"解析失败的回退结果"

### 根因 3：单类型 Prompt 与代码完全脱节（**高优先级**）

`tests/v3_prompts/` 目录下 24 个文件（12 种卡片 × 中英文）**在代码中完全没有被引用**。

`generate_cards` 函数（`cards.rs:224-287`）只加载 `all_cards`：
```rust
let prompt_template = load_prompt("all_cards")?;  // 仅此一个
```

**后果**：
- v3 精心优化的 12 套单类型 Prompt 无法被使用
- 所有卡片由 `all_cards.md` 这一个 Prompt 生成
- `all_cards.md` 是 v1 遗留版本，未同步 v3 升级

### 根因 4：无 JSON Mode 约束（**中优先级**）

卡片生成使用 `call_chat`（文本模式），而非 `call_json`（JSON 模式）：

```rust
// cards.rs:239-253
let response = call_llm.call_chat(...).await;  // 文本模式

// entities.rs:19-20（对比）
let entities = call_llm.call_json(...).await?;  // JSON 模式（有 Schema 约束）
```

文本模式对 LLM 的格式约束力弱，LLM 更容易：
- 忽略规划中的类型分布
- 只输出熟悉的默认格式
- 在遇到格式冲突时"偷懒"

### 根因 5：Token 限制可能过紧（**中优先级**）

```rust
// config.rs:45
pub const MAX_TOKENS_COMPILE: u32 = 8000;
```

8000 tokens 对于生成多类型、多张卡片可能不足。LLM 在达到 token 限制时截断输出，可能只完成 1-2 种类型的卡片。

### 根因 6：NewWord 类型在规划中完全缺失（**低优先级**）

`CardPlanner` 的所有规划函数中均未包含 `CardType::NewWord`：
- `plan_book`（书籍）：无
- `plan_paper`（论文）：无
- `plan_manual`（手册）：无
- `plan_report`（报告）：无
- `plan_article`（文章）：无

即使 Prompt 中列出了新词卡，规划文本也不会推荐生成它。

## 四、v3 Prompts 与源码的脱节分析

### 4.1 脱节清单

| 层面 | v3 Prompts 状态 | 源码状态 | 脱节程度 |
|------|----------------|---------|---------|
| 字段格式 | v3 定义了 12 种卡片的专属字段（如"已知/新知/例子"） | `all_cards.md` 只用"内容"一个字段 | **严重** |
| 输出格式 | v3 要求 `#卡片类型`（Markdown 标题） | 解析器查找 `类型：`（键值对） | **严重** |
| 卡片类型 | v3 新增"技巧卡""任意卡" | 枚举中存在，但 `all_cards.md` 未更新 | **中等** |
| 质量标准 | v3 每种卡片有 6-10 条专属质量标准 | `all_cards.md` 质量标准泛化 | **中等** |
| 角色定义 | v3 每种卡片有独特的"信仰+双重身份" | `all_cards.md` 角色扁平 | **中等** |
| 示例 | v3 每种卡片有 1-3 张完整示例 | `all_cards.md` 示例简单 | **轻微** |
| 引用格式 | v3 统一为"来源名_p页码" | `all_cards.md` 用"当前文档\|章节\|页码" | **轻微** |

### 4.2 核心矛盾

> **v3 单类型 Prompt 的质量无法被使用，因为代码只调用 `all_cards.md`；而 `all_cards.md` 的质量远不及 v3 单类型 Prompt。**

这是架构层面的根本矛盾——"入口 Prompt"（`all_cards.md`）的设计质量决定了整体输出质量，但 `all_cards.md` 没有被同步升级到 v3 标准。

## 五、修复建议（按优先级排序）

### P0 — 立即修复（解决"只生成一种卡片类型"）

#### P0-1：统一 Prompt 格式与解析器格式

**方案 A**（推荐）：修改 `all_cards.md`，将 `#卡片类型` 改为 `类型：卡片类型`

```markdown
---
类型：[卡片类型]
标题：[卡片标题]
内容：[卡片正文内容]
ref：[来源]
uuid：[YYYYMMDDHHMM]
---
```

**方案 B**：修改解析器，同时支持 `#卡片类型` 和 `类型：卡片类型`

```rust
fn parse_card_type(block: &str) -> CardType {
    // 先尝试键值对格式
    if let Some(t) = extract_field(block, "类型") {
        return match_type(t);
    }
    // 再尝试 Markdown 标题格式
    if let Some(t) = extract_markdown_heading(block, "#") {
        return match_type(t);
    }
    CardType::Knowledge
}
```

**建议**：采用方案 A，因为 `all_cards.md` 是 v1 遗留，需要整体重写以匹配 v3 标准。

#### P0-2：重写 `all_cards.md` 以匹配 v3 标准

当前 `all_cards.md` 的核心问题：
1. 用统一的"内容"字段抹平了 12 种卡片的结构差异
2. 角色定义扁平（"知识管理专家"）
3. 质量标准泛化，缺乏类型专属约束
4. 选卡策略降低了多类型生成的强制性

重写方向：
- 在 `all_cards.md` 中保留 v3 的角色定义哲学
- 明确列出 12 种卡片类型的输出格式要求
- 每种类型给出简化的字段模板
- 强调"必须为每种适用的类型生成至少一张卡片"

#### P0-3：接入单类型 Prompt（架构级改造）

**方案**：在 `generate_cards` 中改为"规划 → 分类型调用"模式

```rust
// 伪代码
let plan = CardPlanner::plan(doc_type, document);
let mut all_cards = Vec::new();

for card_type in plan.required_types {
    let prompt = load_prompt(&card_type.prompt_name())?;
    let cards = call_llm.call_chat(prompt).await?;
    all_cards.extend(cards);
}
```

**权衡**：
- 优点：每种卡片使用专属 Prompt，质量最高
- 缺点：API 调用次数增加（一本书可能需要 3-8 次调用）
- 建议：先实现 P0-1 和 P0-2，验证效果后再决定是否接入单类型 Prompt

## 七、评估结论

### 7.1 核心诊断

> **"只生成一种卡片类型"的根本原因是 Prompt 格式与解析器格式不匹配，导致所有卡片被回退为 `CardType::Knowledge`。**

这是一个**格式层面的 bug**，不是 LLM 能力问题。LLM 可能生成了多种类型的卡片，但解析器因为格式不匹配而全部识别失败，统一回退为默认值。

### 7.2 严重程度排序

| 排名 | 问题 | 影响 | 修复难度 |
|------|------|------|---------|
| 1 | `all_cards.md` 格式与解析器不匹配 | 所有卡片变成新知卡 | 低 |
| 2 | v3 单类型 Prompt 未被代码引用 | 高质量 Prompt 无法使用 | 中 |
| 3 | `all_cards.md` 质量远低于 v3 标准 | 即使格式匹配，输出质量仍差 | 中 |
| 4 | 批量处理架构瓶颈（1000 本书） | 无法稳定处理大规模任务 | 中 |
| 5 | `NewWord` 类型在规划中缺失 | 新词卡永远不会被生成 | 低 |
| 6 | 无 JSON Mode 约束 | LLM 格式约束力弱 | 低 |

### 7.3 建议的修复顺序

```
第 1 步：修复格式匹配（P0-1）—— 立即可验证效果
    ↓
第 2 步：重写 all_cards.md（P0-2）—— 提升生成质量
    ↓
第 3 步：接入单类型 Prompt（P0-3）—— 架构升级
    ↓
第 4 步：批量处理优化（P1-1/2/3）—— 支持 1000 本书
    ↓
第 5 步：JSON Schema 化（P2-1）—— 长期质量保障
```
