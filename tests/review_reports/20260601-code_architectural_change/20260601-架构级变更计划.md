## 三、Phase A：独立变更（可并行）

### A1. StageCache 哈希缓存

**问题**：`StageCache::cache_key()` 和 `CompileCache::hash_document()` 各自独立计算 FNV-1a 哈希。同一文档的哈希在 summary/entities/cards/graph 等阶段被重复计算 6-8 次。对 50 万字文档，每次哈希遍历全部字符，累积浪费 ~3-4ms/文档，5000 本 = ~15-20 秒。

**方案**：在 `CompileContext` 中缓存文档哈希，首次计算后所有阶段复用。

**修改文件**：`src/pipeline.rs`

**具体变更**：

1. `CompileContext` 新增字段：
```rust
struct CompileContext {
    // ... 现有字段 ...
    document_hash: Arc<OnceLock<String>>,  // 新增
}
```

2. `CompileContext` 新增方法：
```rust
fn get_doc_hash(&self, document: &str) -> String {
    self.document_hash
        .get_or_init(|| {
            let mut hash: u64 = 0xcbf29ce484222325;
            for byte in document.bytes() {
                hash ^= byte as u64;
                hash = hash.wrapping_mul(0x100000001b3);
            }
            format!("{:x}", hash)
        })
        .clone()
}
```

3. 重构 `try_load_stage()` / `save_stage()` 接受预计算 hash：
```rust
fn try_load_stage<T>(&self, stage: &str, doc_hash: &str, prompt_name: &str) -> Option<T>
fn save_stage<T>(&self, stage: &str, doc_hash: &str, prompt_name: &str, value: &T)
```

4. `StageCache::cache_key()` 接受 `doc_hash: &str` 而非 `document: &str`。

5. `CompileCache::new()` 接受预计算 hash 而非重新计算。

**验收**：
- `cargo test --lib pipeline` 全部通过
- 缓存 hit/miss 行为与修改前一致

**风险**：极低。纯性能优化，无行为变更。

### A3. RPM 限流器（新模块）

**问题**：API 调用间无限流保护。9 次连续调用可能在数秒内完成，触发提供商 RPM 限制（如 DeepSeek 默认 60 RPM）。429 错误 → 3 次重试 → 更频繁触发 → 雪崩。

**方案**：Token-bucket 算法，通过 `CARDNOTE_MAX_RPM` 环境变量配置。未设置时不限流（向后兼容）。

**修改文件**：
- `src/rate_limiter.rs`（**新文件**）
- `src/api.rs`（集成到 `LlmClient`）
- `src/config.rs`（环境变量读取）

**具体变更**：

1. `src/rate_limiter.rs` — 新模块：
```rust
pub struct RateLimiter {
    capacity: u32,
    rate: f64,          // tokens per second
    tokens: f64,
    last_refill: Instant,
}

impl RateLimiter {
    pub fn new(rpm: u32) -> Self { /* rpm / 60 = rps */ }
    pub async fn acquire(&mut self) { /* refill + wait if needed */ }
}
```

2. `src/api.rs` — `LlmClient` 新增字段：
```rust
pub struct LlmClient {
    // ... 现有字段 ...
    rate_limiter: Option<Arc<tokio::sync::Mutex<RateLimiter>>>,
}
```

3. `send_request()` 开始时调用：
```rust

    limiter.lock().await.acquire().await;
}
```

4. `src/config.rs`：
```rust

    std::env::var("CARDNOTE_MAX_RPM").ok().and_then(|s| s.parse().ok())
}
```

**验收**：
- `cargo test --lib rate_limiter`（新增 token-bucket 单元测试）
- 设置 `CARDNOTE_MAX_RPM=5` 验证调用间隔 ~12s
- 不设置时零开销（Option 为 None）

**风险**：低。默认不启用，完全向后兼容。

## 四、Phase B：依赖 Phase A

### B1. JSON 文本降级

**问题**：`try_chat_json_with_model()` 中 JSON 解析失败后重试 3 次（指数退避 2s/4s），全部失败返回 Err。但实际场景中 LLM 经常返回"JSON 包裹在 markdown code fence 中"或"markdown 格式的卡片"——此时直接失败导致该类型卡片全部丢失。

**方案**：3 次 JSON 解析失败后，不直接返回 Err，而是尝试文本降级提取。

**修改文件**：`src/api.rs`

**具体变更**：

1. `send_request()` 存储原始响应文本：
```rust
// 在 send_request 返回前存储
let raw_text = extract_content(&response_body)?;
```

2. 新增降级方法：
```rust
fn degrade_json_to_text(raw: &str) -> Result<Value> {
    // 策略A: 去除 ```json ... ``` code fence 后重试
    if let Some(inner) = extract_code_fence_json(raw) {
        if let Ok(v) = serde_json::from_str(&inner) {
            return Ok(v);
        }
    }
    // 策略B: 对 markdown 卡片格式（含 --- 分隔符），
    // 解析为结构化 card 数组 → 包装为 JSON
    if raw.contains("---") && raw.contains("标题") {
        return parse_cards_as_json(raw);
    }
    // 策略C: 兜底 — 包装为 {"raw_text": "..."}
    Ok(serde_json::json!({"raw_text": raw}))
}
```

3. `try_chat_json_with_model()` 中在所有重试失败后调用降级。
4. 降级成功的卡片设置 `degraded_from = Some(original_type)`。

**验收**：
- 新增单元测试：`degrade_json_to_text` 处理 code fence / markdown / raw 三种场景
- Mock 测试：LLM 返回无效 JSON → 降级成功 → 卡片产出

**风险**：中。降级解析可能脆弱（markdown 格式变化多），但降级卡片质量虽降低，远优于完全丢失。

### B2. Batch CLI（新模块）

**问题**：5000 本 PDF 需手动逐个运行 `cardc file.pdf`。无进度跟踪、无断点续传、无失败重试。

**方案**：SQLite 作业队列 + `cardc batch` CLI 子命令。

**修改文件**：
- `src/batch.rs`（**新文件**）
- `src/main.rs`（新增子命令）
- `Cargo.toml`（新增 `rusqlite` 依赖）

**SQLite Schema**：
```sql
CREATE TABLE IF NOT EXISTS jobs (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    file_path TEXT NOT NULL UNIQUE,
    status TEXT NOT NULL DEFAULT 'pending',
    output_dir TEXT,
    error TEXT,
    summary_title TEXT,
    card_count INTEGER DEFAULT 0,
    prompt_tokens INTEGER DEFAULT 0,
    completion_tokens INTEGER DEFAULT 0,
    started_at TEXT,
    finished_at TEXT,
    retry_count INTEGER DEFAULT 0,
    created_at TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX IF NOT EXISTS idx_jobs_status ON jobs(status);
```

**CLI 接口**：
```bash
cardc batch ./pdfs/                    # 扫描目录，入队所有 PDF，逐条处理
cardc batch ./pdfs/ --resume           # 从上次中断位置继续（只处理 pending）
cardc batch ./pdfs/ --retry-failed     # 只重试状态为 failed 的作业
cardc batch ./pdfs/ --workers 4        # 并发数（默认 1，配合 CARDNOTE_MAX_RPM）
cardc batch ./pdfs/ --skip-existing    # 跳过已有输出目录的文件
cardc batch-status                     # 显示 Pending/Running/Done/Failed 统计
```

**`src/batch.rs` 结构**：
```rust
pub struct BatchRunner {
    db: Connection,              // SQLite（WAL 模式，支持并发读）
    output_dir: PathBuf,
    max_workers: usize,          // Semaphore 控制并发
    skip_existing: bool,
}

impl BatchRunner {
    pub fn new(db_path: &Path, output_dir: &Path, max_workers: usize) -> Result<Self>;
    pub fn populate_jobs(&self, dir: &Path, recursive: bool) -> Result<usize>;
    pub async fn run(&mut self) -> Result<BatchReport>;
    pub async fn retry_failed(&mut self) -> Result<BatchReport>;
    pub fn status(&self) -> Result<BatchStatus>;

pub struct BatchReport {

```

**关键集成点**：
- 复用 `scan::find_pdf_files()` 扫描目录
- 复用 `Pipeline::run()` 处理每文件
- 复用 `output::save_single()` 写输出
- 并发控制：`tokio::sync::Semaphore` + `CARDNOTE_MAX_RPM`

**依赖**：`Cargo.toml` 新增 `rusqlite = { version = "0.31", features = ["bundled"] }`

**验收**：
- 单元测试：创建 3 个极小 PDF，`populate_jobs` → `run` → 3 个输出目录
- 断点续传：杀进程 → `--resume` 从断点继续
- 失败重试：构造损坏 PDF → `--retry-failed` 仅重试失败项
- 状态查询：`batch-status` 输出正确统计
- 内存泄漏：连续处理 100 文件，SQLite 连接不泄漏

**风险**：中。
- `rusqlite` bundled 编译可能在某些平台有问题（bundled feature 自带 SQLite 源码）
- 并发 + 限流需仔细测试，避免死锁

## 五、Phase C：最高影响

### C1. Extract-Then-Assign

**问题**：每本书 9 次顺序 LLM 调用，每次传输完整文档。

**成本量化**（以 scale=2 的 10 万字书籍为例）：

| 项目 | 旧策略（9 次调用） | 新策略（2 次调用） |
|------|-------------------|-------------------|
| 每次输入 token | ~30K（文档+prompt） | ~30K + ~15K（知识点列表） |
| 每次输出 token | ~6K | ~8K |
| 每本总输入 | ~270K | ~45K |
| 每本总输出 | ~54K | ~16K |
| 5000 本总输入 | **1.35B** | **225M** |
| 5000 本总成本 | **¥1,350** | **¥225** |

**方案**：2 次 LLM 调用替代 9 次，失败时自动回退旧策略。

**架构**：
```
generate_cards()
  │
  ├── 文档 ≤ 200K 字符？
  │   ├── 是 → 尝试 generate_cards_extract_then_assign()
  │   │        ├── Step 1: 提取所有知识点（1 次 JSON 调用）
  │   │        └── Step 2: 按类型分配并格式化（1 次文本调用）
  │   │
  │   └── 卡片数 < 必选 min × 50%？→ fallback 旧策略
  │       提取失败？→ fallback 旧策略
  │
  └── 文档 > 200K 字符 → 直接旧策略（避免上下文窗口问题）
```

**修改文件**：
- `src/stages/cards.rs`（核心实现）
- `src/pipeline.rs`（调用签名更新）
- `prompts/extract_knowledge.md`（**新文件**）
- `prompts/assign_cards.md`（**新文件**）

**具体变更**：

1. 重命名 `generate_cards()` → `generate_cards_legacy()`，保持函数体不变。

2. 新增 `generate_cards_extract_then_assign()`：
```rust
pub async fn generate_cards_extract_then_assign(
    document: &str,
    doc_type: DocumentType,
    llm: &(impl ChatFn + ChatJsonFn),
    load_prompt: &(dyn Fn(&str) -> Result<String> + Send + Sync),
) -> Result<Vec<Card>> {
    // Step 1: 提取所有知识点
    let extract_prompt = load_prompt("extract_knowledge")?
        .replace("{document}", &sanitize_for_prompt(document));
    let extraction: Value = llm.call_json(/* ... */).await?;
    let knowledge_points = extraction["knowledge_points"].as_array()?;

    // Step 2: 按类型分配并格式化为卡片
    let assign_prompt = load_prompt("assign_cards")?
        .replace("{knowledge_points}", &json!)
        .replace("{card_plan}", &CardPlanner::plan_text(doc_type, chars));
    let response = llm.call_chat(/* ... */).await?;
    parse_assigned_cards(&response)
}
```

3. 新增 `parse_assigned_cards()`：从 `#卡片类型` 标签检测每张卡片的实际类型。

4. 修改 `generate_cards()` 为混合策略入口：
```rust
pub async fn generate_cards(
    document: &str, doc_type: DocumentType,
    llm: &(impl ChatFn + ChatJsonFn),
    load_prompt: &(dyn Fn(&str) -> Result<String> + Send + Sync),
) -> Result<Vec<Card>> {
    let chars = document.chars().count();
    if chars <= 200_000 {
        match generate_cards_extract_then_assign(document, doc_type, llm, load_prompt).await {
            Ok(cards) if cards.len() >= min_cards_threshold(doc_type, chars) => {
                eprintln!("  ✓ Extract-then-assign: {} cards (2 calls)", cards.len());
                return Ok(cards);
            }
            other => eprintln!("  ⚠ Extract-then-assign fallback: {:?}", other.err()),
        }
    }
    generate_cards_legacy(document, doc_type, llm, load_prompt).await
}
```

5. `min_cards_threshold()` 计算回退阈值：
```rust
fn min_cards_threshold(doc_type: DocumentType, char_count: usize) -> usize {
    CardPlanner::plan(doc_type, char_count)
        .iter()
        .filter(|p| p.required)
        .map(|p| p.min)
        .sum::<usize>() / 2  // 50% 阈值
}
```
