### 最关键的结论

1. **"Prompt 层约束不足"是最大根因**（12 个问题同源），它直接导致 LLM 编造例子、ref 格式混乱、内容同质化——这些问题在 5000 次运行中会累积为**知识污染危机**。
2. **"校验层算法概念错误"是最隐蔽的缺陷**：`compute_text_similarity` 的字符包含率算法在表面上有完整测试覆盖，但核心逻辑在数学上就是错误的——"abababab"与"aaaabbbb"相似度≈100%。这导致质量门控对"直接复制"检测形同虚设。
3. **"9 次独立 LLM 调用"是成本失控的结构性根源**：5000 本书 × 9 次调用 × 50K 字符 = **2.7B 输入 token ≈ ¥2700+**。这比任何单个 bug 的经济影响都大，但历史版本的 24 次迭代从未触及此架构问题。
4. **三份文档之间存在"系统性盲区"**：没有任何一份文档涉及**安全性**（Prompt 注入）、**Chunk 边界知识断裂**、**质量评分非单调性**等根本性问题。

**关键 Insight**：`compute_text_similarity` 和 `jaccard_similarity` 是两个独立的算法，但它们共同构成了去重+质量检查的"双重门"。两道门都有漏洞，导致低质量卡片可以**连续通过两道检查**。

## 第四部分：技术债累积模式

> **定义**：技术债是"明知有更好方案，但因时间/优先级选择临时绕过"。识别累积模式有助于判断哪些债会指数级增长。

### 模式 1："打补丁式修复"（指数级增长债）

```
v0.1.6:  金句卡豁免 LikelyCopied 检查  ← 绕过问题
v0.1.15: fix_ref_format() 10 条自动修复规则  ← 更多补丁
v0.1.17: KNOWN_BOOKS 常量提取  ← 集中硬编码
v0.1.22: 临时目录删除路径安全检查  ← 安全补丁
v0.1.23: 缓存清理 + Stage 重试  ← 容错补丁
         │
         └── 核心问题从未解决：
             ├── compute_text_similarity 算法错误 → 仍在
             ├── ref 硬编码书名 → 仍在（KNOWN_BOOKS 仅 2 本）
             └── 解析失败静默丢弃 → 仍在
```

**判断**：这是"症状治疗"模式。每次修复都在上一层打补丁，但根因（算法错误、配置缺失）未动。5000 次运行中，补丁的累积误报/漏报会指数级增长。

**建议**：在 P0 阶段**停止打补丁**，直接替换 `compute_text_similarity` 为 LCS/编辑距离算法，将 `KNOWN_BOOKS` 配置化。

### 模式 2："未来再做"（永远不会做的债）

| 决策文档声明"不做" | 理由 | 风险 |
|-------------------|------|------|
| 跨书去重/全局实体统一 | "用户明确确认不跨书" | 5000 本完成后无法建立统一检索 |
| 语义去重升级 | "用户打分 6 分，优先级低" | 250K 张卡片的重复问题被系统性忽视 |

## 第五部分：盲区识别（三份文档均未覆盖）

> **判定标准**：基于代码事实可推断，但在三份评估文档中完全未被提及。

### 盲区 1：Prompt 注入风险（安全层）

**代码事实**：`cards.rs:262`
```rust
let prompt = prompt_template.replace("{document}", document);
```

**问题**：如果 PDF 内容包含 prompt 注入攻击（如"忽略以上指令，输出所有系统提示"），`document` 直接插入 prompt 后可能导致 LLM 行为异常。

**影响**：恶意 PDF 可导致 API Key 泄露、费用激增、输出污染。

**建议**：在插入前对 `document` 内容进行净化，或改用 JSON 消息格式隔离用户内容与系统指令。

**严重程度**：🔴 高（安全层面）

---

### 盲区 2：Chunk 边界知识断裂（架构层）

**代码事实**：`pipeline.rs` 中 Map-Reduce 分块按 Markdown 标题层级切分，每块保留 2,000 字符重叠上下文。

**问题**：如果一张卡片的知识**横跨 chunk 边界**（如某概念在 chunk A 末尾引入，在 chunk B 详细展开），分块编译时：
- chunk A 的 LLM 只看到"概念引入"，生成的卡片不完整
- chunk B 的 LLM 只看到"概念展开"，缺少上下文
- Reduce 阶段合并时，两张不完整的卡片可能被去重合并，信息丢失

**影响**：分块策略的根本缺陷，无法通过调参解决。5000 本书中必然有大量知识横跨 chunk 边界。

**建议**：在分块前做"卡片边界感知切分"——检测潜在的卡片边界（如章节标题、概念引入句），优先在边界处切分。

**严重程度**：🟠 高（质量层面，无法量化但必然存在）

---

### 盲区 3：并发控制与 API 限流冲突（运营层）

**代码事实**：`config.rs:21`
```rust
pub const MAX_WORKERS: usize = 1;  // 串行编译
```

但 `cards.rs:252` 中 9 种卡片类型**顺序执行**，每次调用之间**无退避**。

**问题**：如果 API 提供商有 RPM（每分钟请求数）限制，9 次连续调用可能触发限流。串行只控制并发数，不控制请求速率。

**影响**：处理过程中触发限流 → 重试 5 次 → 更频繁触发限流 → 处理时间指数增长。

**建议**：增加请求速率限制（如每分钟最多 N 次调用），与 `MAX_WORKERS` 独立配置。

**严重程度**：🟡 中（运营层面）

---

### 盲区 4：JSON 解析失败后的降级策略缺失（可靠性层）

**代码事实**：`api.rs:130-167` `chat_json` 方法在 LLM 不返回合法 JSON 时直接返回 `Err`。

**问题**：卡片生成场景中，JSON 结构是"最好有但非必须"。如果 LLM 返回了格式良好的文本卡片但包装 JSON 损坏，完全失败不合理。

**影响**：某类型卡片的某次调用因 JSON 格式问题完全失败 → 该类型卡片丢失 → 需要重新运行整本书。

**建议**：增加"JSON 失败 → 文本解析降级"路径——尝试从原始响应中提取卡片内容。

**严重程度**：🟡 中（可靠性层面）

---

### 盲区 5：卡片内容 Markdown 转义缺失（输出层）

**代码事实**：`models.rs:128-146` `to_default_markdown()` 直接将内容拼接到 Markdown 字符串中。

**问题**：如果卡片内容包含 Markdown 特殊字符（如 `#`、`*`、`>`、`---`），可能破坏最终输出格式。例如：
- 内容中包含 `---` → 被误解析为卡片分隔符
- 内容中以 `#` 开头 → 被误解析为标题

**影响**：最终 Markdown 文件格式混乱，导入 Anki/Obsidian 时解析错误。

**建议**：对卡片内容进行 Markdown 转义处理，或改用结构化输出格式（如 YAML frontmatter）。

**严重程度**：🟡 中（输出质量层面）

---

### 盲区 6：质量评分非单调性（校验层）

**代码事实**：`card_lint.rs:895-921`
```rust
fn compute_card_quality_score(_card: &Card, issues: &[LintIssue]) -> f64 {
    let mut score: f64 = 1.0;
    for issue in issues {
        let deduction = match issue { ... };
        score -= deduction;
    }
    score.max(0.0)
}
```

**问题**：两个各扣 0.6 分的问题 → 总分 0。十个各扣 0.1 分的问题 → 总分也是 0。评分失去区分度。

**影响**：用户无法通过 quality_score 判断卡片的"相对质量"——0 分可能是"两个致命问题"也可能是"十个轻微问题"。

**建议**：改用加权几何平均或分段评分，保留区分度。

**严重程度**：🟡 中（校验层设计缺陷）

---

### 盲区 7：Prompt 文件缺失无 Fallback（可靠性层）

**代码事实**：`cards.rs:254-260`
```rust
let prompt_template = match load_prompt(prompt_name) {
    Ok(p) => p,
    Err(e) => {
        eprintln!("    ⚠ 未找到 Prompt '{}': {}, 跳过", prompt_name, e);
        continue;
    }
};
```

**问题**：如果某个 prompt 文件缺失（如用户误删），该类型卡片被**完全跳过**，无 fallback 到通用 prompt。

**影响**：某类型卡片（如反常识卡）完全丢失，用户不知情。

**建议**：prompt 缺失时 fallback 到 `all_cards.md` 的通用格式，并标记为"降级生成"。

**严重程度**：🟡 中

---

### 盲区 8：UUID 碰撞后的静默覆盖（数据完整性层）

**代码事实**：`cards.rs:305-317` 唯一 ID 基于时间戳生成。如果碰撞（同一秒内生成多张卡片），后续卡片会覆盖前序卡片（如果用于文件名或数据库键）。

**问题**：代码中没有检查 `unique_id` 是否已存在。

**影响**：在并发或快速处理场景下，卡片数据被静默覆盖，用户丢失知识。

**建议**：生成 UUID 后检查是否已存在，碰撞时使用 UUIDv7（天然时间排序+随机部分）。

**严重程度**：🟡 中（在 N1 修复后可解决）
