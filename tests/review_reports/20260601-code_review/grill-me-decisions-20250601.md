## 二、目标与约束层

### 2.1 总目标

将 5000 本各个领域的经典书籍 PDF 编译为高质量结构化知识卡片（Markdown 格式），基于阳志平卡片大法，输出 12 种标准卡片类型。

### 2.2 处理模式：Craftsman（工匠模式）

| 维度 | 选择 | 说明 |
|------|------|------|
| 批量 vs 单本 | **单本** | 一本一本手动处理，不追求自动化吞吐 |
| 速度 vs 质量 | **质量优先** | 不赶进度，每张卡片都达到验收标准 |
| 全自动 vs 人工介入 | **混合** | 编译全自动，人工审阅产出物 |

**关键约束**：5000 本是总量目标，但处理单元始终为"单本书"（一次只处理一本书）。不存在"批量编排架构"需求。Map-Reduce 路径中的"最多 2 并发"指单书内的分块并行度，非多书并发。

### 2.3 消费场景

当前阶段**仅专注卡片生产**，输出为通用 Markdown 格式。后续消费端（Anki / Obsidian / Logseq / 检索库）作为独立阶段开发，不影响当前卡片生产管线。

### 2.4 不跨书处理

每本书独立编译，独立产出。不建立跨书全局知识图谱，不进行跨书实体统一或去重。

## 三、质量标准层

### 3.1 质量维度打分（用户确认）

| 维度 | 打分 | 权重解读 | 当前机制 | 改进决策 |
|------|------|---------|---------|---------|
| **内容原创性** | **10** | 最高优先级 | Prompt 约束"用自己的话写，不要复制原文" | 依赖 LLM 遵循，无额外自动验证 |
| **源头可追溯** | **10** | 最高优先级 | `ref: 来源名_p页码` + `uuid` | **→ 新增 L2 文本片段模糊匹配验证** |
| **排版质量** | **8** | 高优先级 | typo_lint（中西文间距、标点挤压、直角引号等） | 保持当前 |
| **信息密度** | **7** | 中优先级 | Prompt 限制 200-500 字/卡 | **暂不扩展**（用户明确"暂时先不改"） |
| **类型准确性** | **7** | 中优先级 | 12 种卡片类型 + CardPlanner 动态规划 | 保持当前 |
| **LLM 幻觉过滤** | **7** | 中优先级 | 无自动验证 | **→ L2 引用验证间接覆盖** |
| **跨 chunk 去重** | **6** | 低优先级 | 简单 `(title, type)` 去重 | 保持当前 |
| **语义去重** | **6** | 低优先级 | 未实现 | 保持当前 |

### 3.2 质量验收方式

Craftsman 模式下，质量验收依赖两层机制：

1. **自动层**：L2 引用验证 + 排版 lint + 空卡片过滤 + card_quality_report.md
2. **人工层**：用户逐本审阅卡片产出，依托质量报告判断

**失败卡片处理**：当前版本不自动修复或丢弃，仅标记状态并输出到 `card_quality_report.md`，由用户在 craftsman 审阅环节处理。后续版本再讨论自动化策略。

## 四、编译策略层：混合策略（Hybrid Strategy）

### 4.1 策略概述

```
输入文档
  │
  ▼
[字符数判断]
  ├─ ≤ 120,000 字符 ──→ [长上下文单轮编译] ──→ 质量检测 ──→ 达标？→ 产出
  │                                              │
  │                                              └─ 不达标 ──→ [Fallback: Map-Reduce 分块]
  │
  └─ > 120,000 字符 ───→ [直接 Map-Reduce 分块编译] ──→ 产出
```

### 4.2 长上下文单轮编译（路径 A）

**适用条件**：文档字符数 ≤ 120,000（原 CHUNK_SIZE 从 50,000 上调。选取 120K 的依据：Kimi 等主流长上下文模型在 100K–150K 字符区间内注意力衰减尚可接受，且 120K 约对应一本 200 页左右的中文书籍，覆盖多数常见场景。阈值后续可根据实际运行数据校准）。

**执行方式**：
- 使用 Kimi 2.6（长上下文模型）一次性传入全书
- 串行执行 4 个阶段：摘要 → 实体 → 卡片 → 图谱
- 每个阶段有独立重试机制（见 6.1 节）

**优势**：
- 全局一致性最佳，无分块边界信息丢失
- 卡片间关联更准确
- 单本调用次数极少

**风险**：
- "Lost in the Middle"：模型对长文本中间部分注意力衰减
- 输出 token 上限（通常 8K）可能截断卡片列表

### 4.3 Map-Reduce 分块编译（路径 B）

**适用条件**：文档字符数 > 120,000，或路径 A fallback 触发。

**执行方式**：
- 语义分块（按 Markdown 标题层级，每块 ≤ 120,000 字符）
- 块间保留 2,000 字符重叠上下文
- 每块串行编译（最多 2 并发，避免 API 限流）
- Reduce 阶段合并去重 + 全局摘要
- 支持断点续编译（`.cardc_cache/`，Phase 2 计划实现）

### 4.4 Fallback 触发条件（全自动）

单轮编译完成后，系统自动检测以下任一条件，触发 fallback 到 Map-Reduce：

| 条件 | 检测方式 | 说明 |
|------|---------|------|
| **A. 输出为空** | 摘要标题+概述全空，且卡片数量为 0 | 说明 LLM 完全未产出 |
| **B. 输出截断** | LLM 返回 `finish_reason = "length"` | 输出 token 达到上限 |
| **C. 产出不足** | 实际卡片数 < 规划数量的 50% | 说明 LLM 遗漏大量内容 |
| **D. 阶段失败** | 任一阶段 5 次重试后仍失败 | 网络/API/模型异常 |

**Fallback 执行**：自动切换到 Map-Reduce 分块编译，使用 Tiered 模型分配（见第五节）。

## 五、模型分配层（Tiered Strategy）

### 5.1 模型能力特征

> **注意**：下表中的模型名称指目标配置版本，与当前 `providers.rs` 中的存量模型 ID 可能不完全一致。具体模型 ID 以实际部署时的 provider 配置为准。

| 模型 | 核心优势 | 适用场景 |
|------|---------|---------|
| **Kimi 2.6** | 超长上下文（200K+ tokens） | 需要完整文档输入的阶段 |
| **DeepSeek V4 Pro** | 推理能力最强 | 需要深度理解、提炼、整合的阶段 |
| **MiniMax M2.7** | 通用能力 | 备用 fallback，主模型不可用时切换 |

### 5.2 阶段-模型映射

| 编译路径 | 阶段 | 模型 | 理由 |
|---------|------|------|------|
| **单轮编译**（短书） | 全部阶段 | **Kimi 2.6** | 长上下文一次吞下全书，保证全局一致性 |
| **分块编译**（长书） | 摘要 | **DeepSeek V4 Pro** | 强推理，精准提炼核心论点 |
| | 实体提取 | **Kimi 2.6** | 实体提取依赖 chunk 内完整上下文，Kimi 在结构化抽取任务上格式遵循度高 |
| | 卡片生成 | **Kimi 2.6** | 长上下文 + 格式遵循能力 |
| | 图谱构建 | **Kimi 2.6** | JSON 输出 + 关系推理 |
| | Reduce 合并 | **DeepSeek V4 Pro** | 全局摘要合并需要最强推理 |
| **全局 Fallback** | 任意阶段不可用 | **MiniMax M2.7** | 备用切换，保证管线不中断 |

### 5.3 配置方式

通过环境变量或 `.env` 配置：

```bash
# 默认模型（单轮编译用）
LLM_MODEL=kimi-2.6

# 分块编译的 Tiered 分配
LLM_MODEL_SUMMARY=deepseek-v4-pro
LLM_MODEL_ENTITIES=kimi-2.6
LLM_MODEL_CARDS=kimi-2.6
LLM_MODEL_GRAPH=kimi-2.6

# 备用模型（自动 fallback）
LLM_FALLBACK_MODEL=minimax-m2.7
```

## 六、超时与容错层

### 6.1 重试策略

| 参数 | 原值 | 新值 | 说明 |
|------|------|------|------|
| 重试次数 | 3 次 | **5 次** | 长上下文调用不稳定，增加容错 |
| 退避间隔 | 2s→4s→8s | **5s→10s→20s→40s→80s** | 更长退避，避免触发 API 限流 |
| 超时类型 | 统一超时 | **连接超时 + 读取超时分离** | 长上下文读取时间差异大 |

### 6.2 阶段容错（Stage Tolerance）

**分块编译**中的阶段容错策略：

```
摘要阶段 ──→ 成功/失败 ──→ 继续实体阶段（失败则摘要为空）
实体阶段 ──→ 成功/失败 ──→ 继续卡片阶段（失败则实体为空）
卡片阶段 ──→ 成功/失败 ──→ 继续图谱阶段（失败则卡片为空）
图谱阶段 ──→ 成功/失败 ──→ 完成（失败则图谱为空）
```

**原则**：阶段失败不阻塞下游，记录诊断信息到 `compile_diagnostics.md`（Phase 2 新增），继续最大化产出。

**单轮编译**保持当前行为：全有或全无（因用户确认"单轮不需要分阶段独立重试"）。

### 6.3 分块编译失败率阈值

保持当前行为：失败率 > 20% 时报错中断，不产出降级结果。

**理由**：Craftsman 模式下，用户倾向于"全好或全坏"，不接受"半吊子"产出。

## 七、引用验证层（新增）

### 7.1 问题背景与验证分层

**验证分层定义**：

| 层级 | 名称 | 验证内容 | 状态 |
|------|------|---------|------|
| L1 | 格式层验证 | 正则检查 `ref` 字段合法性（书名、页码格式、禁止前导零等） | 已有 |
| L2 | 内容层验证 | 文本片段模糊匹配，验证卡片内容确实来自 ref 标注的原文位置 | **本次新增** |

"源头可追溯"是用户最高优先级（10 分），但当前系统仅依赖 LLM 自律输出 `ref` 字段，L1 格式验证无法检测内容层面的引用错误。LLM 可能：
- 编造不存在的页码
- 张冠李戴（引用 A 段落的内容标注 B 页码）
- 输出格式不规范的 ref
- 在 L1 格式合法的前提下伪造看似合理的引用

### 7.2 L2 文本片段模糊匹配方案

**验证流程**：

```
卡片生成完成
  │
  ▼
提取卡片关键短语（3-5 个核心词，使用基于 TF-IDF 的关键词提取或直接取卡片标题/首句中的名词短语）
  │
  ▼
定位 ref 标注的页码范围（在保留的原始 PDF 文本中定位）
  │
  ▼
在该页码附近做模糊匹配（编辑距离 / 子串包含）
  │
  ├─ 匹配成功 ──→ 标记 status: Accepted
  └─ 匹配失败 ──→ 标记 status: NeedsRetry，写入质量报告
```

**技术参数**：
- 匹配算法：基于子串包含 + 编辑距离阈值（Jaro-Winkler ≥ 0.6）
- 搜索范围：ref 标注页码 ± 2 页
- 性能：纯本地计算，零额外 API 成本
- 容错：因要求"用自己的话写"，原文与卡片内容不会完全匹配，阈值需足够宽容

### 7.3 失败处理

| 场景 | 处理方式 |
|------|---------|
| 匹配成功 | 卡片状态保持 `Accepted` |
| 匹配失败 | 卡片状态标记为 `NeedsRetry`，原因写入 `reject_reason`，在 `card_quality_report.md` 中标注 |
| ref 格式非法 | 直接标记为 `Rejected` |

**注意**：当前版本不自动修复或丢弃失败卡片，由用户在 craftsman 审阅环节处理。
